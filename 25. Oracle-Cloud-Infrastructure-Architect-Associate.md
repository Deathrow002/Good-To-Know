# Oracle Cloud Infrastructure 2023 Certified Architect Associate — Interview Questions and Answers

This guide contains original interview questions and concise answers aligned with **Oracle Cloud Infrastructure 2023 Certified Architect Associate (1Z0-1072-23)** skills. It covers OCI core architecture, IAM, networking, compute, storage, databases, high availability and disaster recovery, security, observability, and cost governance.

> **Note:** This is an interview-preparation guide, not an exam dump. OCI services and certification objectives evolve, and behavior can differ by region and tenancy. Verify current limits and procedures in official Oracle documentation.

---

## 1. OCI Core Architecture and Foundations

### Q1. What is a region in OCI?

**Answer:** A **region** is a localized geographic area containing one or more data centers. It is the top-level unit for deploying resources and often maps to data residency and latency requirements. Some services are region-scoped while others are availability-domain-scoped.

### Q2. What is an availability domain?

**Answer:** An **availability domain (AD)** is one or more isolated data centers within a region with independent power, cooling, and network. ADs in a region do not share a single point of failure, so distributing resources across ADs improves availability.

### Q3. What is a fault domain?

**Answer:** A **fault domain (FD)** is a grouping of hardware within a single availability domain. Each AD typically has three fault domains. Placing instances in different fault domains protects against hardware failure and some maintenance events within one AD.

### Q4. How do regions, availability domains, and fault domains relate?

**Answer:** A region contains availability domains, and each availability domain contains fault domains. Resilient designs spread redundant components across fault domains within an AD, and across availability domains or regions for stronger isolation where the services support it.

### Q5. What is a single-availability-domain region?

**Answer:** Some regions contain only one availability domain. In these regions, high availability is achieved by distributing resources across **fault domains**, and disaster recovery is achieved by using a second region.

### Q6. What is a tenancy?

**Answer:** A **tenancy** is the root container for all of an organization’s OCI resources, created when the account is provisioned. It is the top of the compartment hierarchy and the boundary for billing and top-level administration.

### Q7. What is a compartment?

**Answer:** A **compartment** is a logical container for organizing and isolating OCI resources. Compartments enable access control through policies, cost tracking, and separation of environments or teams. They are global and can be nested.

### Q8. Is a compartment a network boundary?

**Answer:** No. A compartment is an authorization and organization boundary, not a network boundary. Network isolation is provided by virtual cloud networks, subnets, and security rules. Resources in different compartments can communicate if the network and policies allow it.

### Q9. What is an OCID?

**Answer:** An **Oracle Cloud Identifier (OCID)** is a globally unique identifier assigned to every OCI resource. It is used in policies, APIs, CLI commands, and automation to reference a specific resource unambiguously.

### Q10. How should compartments be structured for an enterprise?

**Answer:** Structure compartments around environments, business units, or workloads to support least-privilege policies, cost visibility, and governance. Common patterns separate production from non-production and place shared network or security resources in dedicated compartments.

> **Interview tip:** Remember that a compartment controls *who can manage* resources, while a VCN controls *what can talk to* what.

## 2. Identity and Access Management

### Q11. What are the main building blocks of OCI IAM?

**Answer:** The main building blocks are **principals** (users and resources), **groups**, **compartments**, **policies**, and **resources**. Policies grant groups or dynamic groups permission to perform verbs on resource types within a compartment or the tenancy.

### Q12. What is the structure of an IAM policy statement?

**Answer:** A common form is:

```text
Allow group <group-name> to <verb> <resource-type> in compartment <compartment-name>
```

Optional `where` conditions can further restrict the statement. Policies are allow-only; there are no explicit deny statements.

### Q13. What are the IAM policy verbs?

**Answer:** From least to most permissive, the verbs are generally `inspect`, `read`, `use`, and `manage`. Each verb maps to a set of permissions that depends on the resource type. Prefer the least verb that satisfies the requirement.

### Q14. What is the difference between a group and a dynamic group?

**Answer:** A **group** contains IAM users. A **dynamic group** contains OCI resources such as compute instances or functions, identified by matching rules. Dynamic groups enable resources to call OCI APIs through instance or resource principals without stored user credentials.

### Q15. What is an instance principal?

**Answer:** An **instance principal** lets a compute instance authenticate to OCI services using its identity rather than an API key stored on the instance. The instance is added to a dynamic group, and policies grant the required permissions. This avoids embedding long-lived credentials.

### Q16. How is federation used in OCI?

**Answer:** **Federation** links an external identity provider to OCI so users can sign in with existing corporate credentials. OCI supports SAML-based federation and integration with Oracle-managed identity domains, centralizing authentication and reducing separate account management.

### Q17. What is the difference between authentication and authorization?

**Answer:** **Authentication** verifies who a principal is. **Authorization** determines what that principal can do, evaluated by IAM policies. A successfully authenticated user still needs matching policies to perform an action.

### Q18. What are common credential types in OCI?

**Answer:** Common credentials include console passwords, API signing keys, auth tokens (for services such as the registry), and secret keys for S3-compatible access. Each type should be scoped, rotated, monitored, and revoked when no longer needed.

### Q19. How do you enforce least privilege in OCI?

**Answer:** Grant policies at the narrowest compartment scope, use the least permissive verb, separate duties across groups, use conditions where helpful, prefer dynamic groups and resource principals for workloads, and review access regularly. Avoid granting broad tenancy-level `manage all-resources` except for a small administrative group.

### Q20. What does the `manage all-resources` policy do?

**Answer:** It grants full management of all resource types in the specified scope. At the tenancy level it is effectively administrator access, so it should be limited to a small, protected administrator group and audited.

## 3. Networking and Virtual Cloud Networks

### Q21. What is a Virtual Cloud Network?

**Answer:** A **Virtual Cloud Network (VCN)** is a customizable private network in an OCI region. It has a CIDR range and contains subnets, route tables, gateways, security lists, and network security groups that control connectivity.

### Q22. What is the difference between a public and a private subnet?

**Answer:** Instances in a **public subnet** can have public IP addresses and reach the internet through an internet gateway. Instances in a **private subnet** have only private IP addresses and reach external services through a NAT gateway or service gateway. Private subnets reduce exposure.

### Q23. Can a subnet span availability domains?

**Answer:** Subnets can be **regional** (spanning all availability domains in the region) or AD-specific. Regional subnets are generally recommended because they simplify high-availability design and allow resources to move within the region.

### Q24. What are the main VCN gateways?

**Answer:**

| Gateway | Purpose |
|---|---|
| Internet Gateway | Bidirectional internet access for public subnets |
| NAT Gateway | Outbound-only internet access for private resources |
| Service Gateway | Private access to supported OCI services |
| Dynamic Routing Gateway | Connectivity to on-premises or other VCNs |
| Local Peering Gateway | Peering between VCNs in the same region |

### Q25. What is the difference between a security list and a network security group?

**Answer:** A **security list** applies rules to all VNICs in a subnet. A **network security group (NSG)** applies rules to a specific set of VNICs regardless of subnet. NSGs enable finer, workload-based segmentation and can reference other NSGs as sources or destinations.

### Q26. Are OCI security rules stateful or stateless?

**Answer:** They can be either. **Stateful** rules automatically allow return traffic and are the common default. **Stateless** rules require explicit rules for both directions and can offer performance benefits for high-throughput workloads, but need careful bidirectional configuration.

### Q27. What is a route table used for?

**Answer:** A **route table** directs traffic leaving a subnet toward the appropriate gateway or target based on destination CIDR. For example, a private subnet route table might send internet-bound traffic to a NAT gateway and OCI-service traffic to a service gateway.

### Q28. What is a Dynamic Routing Gateway?

**Answer:** A **Dynamic Routing Gateway (DRG)** is a virtual router that connects a VCN to on-premises networks through VPN or FastConnect and can connect VCNs across regions. It is central to hybrid and multi-VCN connectivity.

### Q29. What is the difference between Site-to-Site VPN and FastConnect?

**Answer:** **Site-to-Site VPN** creates encrypted tunnels over the public internet. **FastConnect** provides a private, dedicated connection with more consistent bandwidth and latency. VPN is faster to set up, while FastConnect suits high or predictable throughput and stricter requirements.

### Q30. What is local VCN peering versus remote VCN peering?

**Answer:** **Local peering** connects two VCNs in the same region using a local peering gateway. **Remote peering** connects VCNs in different regions using DRGs. Both require non-overlapping CIDRs and appropriate route and security configuration.

### Q31. Why must peered VCNs have non-overlapping CIDRs?

**Answer:** Overlapping address ranges make routing ambiguous, so traffic cannot be reliably directed to the correct destination. Address planning should reserve distinct CIDR ranges for VCNs that may need to communicate.

### Q32. What is a service gateway and why is it useful?

**Answer:** A **service gateway** lets resources in a VCN privately reach supported OCI services, such as Object Storage, without traversing the internet. It keeps traffic on the OCI network, improving security and avoiding the need for a NAT gateway for those services.

### Q33. What are the main OCI load balancer options?

**Answer:** OCI offers a **Load Balancer** operating at layer 7 (HTTP/HTTPS) with features such as path routing and SSL handling, and a **Network Load Balancer** operating at layer 4 with high performance and low latency. The choice depends on protocol, features, and performance needs.

### Q34. What is a backend set in a load balancer?

**Answer:** A **backend set** is a group of backend servers plus a load-balancing policy and a health-check policy. The load balancer distributes traffic among healthy backends according to the policy, such as round robin, least connections, or IP hash.

### Q35. How does a load balancer health check improve availability?

**Answer:** It periodically tests each backend and removes unhealthy ones from rotation, so traffic is sent only to responsive servers. Combined with backends across fault or availability domains, it maintains service during individual failures.

### Q36. How do you design a highly available network tier?

**Answer:** Use a regional subnet, distribute backends across fault domains or availability domains, front them with a load balancer, apply least-privilege NSGs, plan redundant connectivity such as multiple VPN tunnels or FastConnect, and avoid single points of failure in routing.

## 4. Compute

### Q37. What compute options does OCI provide?

**Answer:** OCI offers **virtual machine** instances, **bare metal** instances, and **dedicated** options. VMs share physical hardware with strong isolation, bare metal provides an entire physical server, and dedicated hosts allow running VMs on isolated hardware for compliance or licensing.

### Q38. What is a compute shape?

**Answer:** A **shape** defines the CPU, memory, and other resources of an instance. **Flexible shapes** let you choose the number of OCPUs and amount of memory within supported ranges, enabling right-sizing.

### Q39. What is an OCPU?

**Answer:** An **OCPU** represents CPU capacity. Depending on the processor, one OCPU may correspond to a physical core and its associated hardware threads. Sizing should be based on measured workload behavior rather than assumptions.

### Q40. What is a custom image?

**Answer:** A **custom image** captures an instance’s configured operating system and software so new instances can be launched from a known baseline. It supports standardization, faster provisioning, and consistent deployments.

### Q41. What is an instance configuration and instance pool?

**Answer:** An **instance configuration** is a template describing how to create instances. An **instance pool** creates and manages a group of identical instances from that configuration across fault or availability domains, enabling consistent scaling and management.

### Q42. What is autoscaling in OCI compute?

**Answer:** **Autoscaling** adjusts the number of instances in a pool based on metrics such as CPU or memory (metric-based) or on a schedule (schedule-based). It helps match capacity to demand while controlling cost. Workloads should be stateless for smooth scaling.

### Q43. What is the difference between vertical and horizontal scaling for compute?

**Answer:** **Vertical scaling** changes an instance’s shape to add or remove CPU and memory, often requiring a restart. **Horizontal scaling** adds or removes instances behind a load balancer. Horizontal scaling generally provides better availability and elasticity for cloud-native workloads.

### Q44. What is a boot volume?

**Answer:** A **boot volume** is the block storage volume containing the operating system for a compute instance. It persists independently of the instance lifecycle in supported cases and can be backed up and restored.

### Q45. How do you place compute for high availability within a single AD?

**Answer:** Distribute instances across the availability domain’s **fault domains**, use an instance pool or manual placement, and front them with a load balancer. This protects against localized hardware and maintenance failures without requiring multiple availability domains.

## 5. Storage

### Q46. What are the main OCI storage services?

**Answer:** The main services are **Block Volume** for persistent disk attached to instances, **Object Storage** for unstructured data accessed over HTTP APIs, **File Storage** for shared network file systems, and **Archive Storage** for low-cost, rarely accessed data.

### Q47. What is Block Volume?

**Answer:** **Block Volume** provides network-attached, durable block storage that can be attached to compute instances. Volumes can be resized, backed up, cloned, and moved between instances. They are commonly used for databases and application data requiring low-latency disk access.

### Q48. What is the difference between block, object, and file storage?

**Answer:**

| Type | Access model | Typical use |
|---|---|---|
| Block Volume | Attached as a raw disk to one instance | Databases, boot volumes, low-latency I/O |
| Object Storage | HTTP API with buckets and objects | Backups, media, data lakes, static content |
| File Storage | Shared network file system (NFS) | Shared access across many instances |

### Q49. What is the difference between the Standard and Archive Object Storage tiers?

**Answer:** **Standard** storage is for frequently or immediately accessed data with low retrieval latency. **Archive** storage is much cheaper for long-term retention but requires a restore step and wait time before objects can be read, making it suitable for rarely accessed data.

### Q50. What is a pre-authenticated request in Object Storage?

**Answer:** A **pre-authenticated request (PAR)** grants time-limited access to a bucket or object without requiring the recipient to have OCI credentials. It is useful for controlled uploads or downloads, but the URL must be treated as a secret because anyone with it has the granted access.

### Q51. How does Object Storage provide durability?

**Answer:** Object Storage stores data redundantly within the region and uses integrity checks to detect and repair corruption. Optional cross-region replication can further protect against regional loss. It is designed for high durability of stored objects.

### Q52. What is the difference between a volume backup and a volume clone?

**Answer:** A **backup** is a point-in-time copy stored separately and used to restore a new volume later. A **clone** is an immediate, independent copy of a volume that can be used right away. Backups suit retention and recovery; clones suit fast duplication for testing or scaling.

### Q53. What is volume group backup used for?

**Answer:** A **volume group** lets you manage and back up multiple related volumes together, producing a consistent set of backups. This is useful for multi-volume applications where the volumes must be recovered as a coordinated unit.

### Q54. How is data encrypted in OCI storage?

**Answer:** OCI encrypts data at rest by default. Keys can be Oracle-managed or customer-managed through OCI Vault. Customer-managed keys give control over rotation and revocation. Data in transit can be protected with TLS.

### Q55. How do you choose a storage service for an application?

**Answer:** Match the access pattern: use Block Volume for low-latency disk on a single instance, File Storage for shared file access across instances, Object Storage for scalable API-based unstructured data, and Archive for infrequently accessed retention. Consider durability, performance, cost, and access model.

## 6. Databases on OCI

### Q56. What database options does OCI provide?

**Answer:** OCI offers **Autonomous Database**, **Base Database Service**, **Exadata Database Service**, and the ability to run databases on compute. Options differ in automation, control, performance, and cost, from fully managed autonomous to self-managed on compute.

### Q57. What is Autonomous Database?

**Answer:** **Autonomous Database** is a fully managed service that automates provisioning, patching, tuning, scaling, and backups. It comes in workload types such as transaction processing and data warehouse, and reduces routine administration compared with co-managed services.

### Q58. What is the difference between Autonomous Database and Base Database Service?

**Answer:** **Autonomous Database** automates most administration and limits operating-system access. **Base Database Service** is co-managed, giving the customer more control over the database, operating system, and configuration while OCI automates common lifecycle operations. The choice depends on control versus automation needs.

### Q59. What is Exadata Database Service?

**Answer:** **Exadata Database Service** runs Oracle Database on Exadata infrastructure optimized for high performance, scalability, and availability. It suits demanding or consolidated workloads requiring Exadata capabilities.

### Q60. How are database backups typically handled on OCI?

**Answer:** Managed database services support automatic backups to OCI-managed destinations with configurable retention. Administrators should still monitor backup success, test restores, and align retention with recovery objectives rather than assuming backups guarantee recoverability.

### Q61. How do you make a database highly available on OCI?

**Answer:** Use options such as RAC for instance-level availability where supported, Data Guard for standby-based protection across fault domains, availability domains, or regions, resilient storage, tested backups, and application connection strategies that tolerate failover.

## 7. High Availability and Disaster Recovery

### Q62. What is the difference between high availability and disaster recovery?

**Answer:** **High availability (HA)** keeps a service running through localized failures, usually within a region. **Disaster recovery (DR)** restores service after a larger failure, typically by using a second region. HA reduces downtime; DR addresses regional or catastrophic events.

### Q63. What is the difference between RPO and RTO?

**Answer:** **Recovery Point Objective (RPO)** is the maximum acceptable data loss measured in time. **Recovery Time Objective (RTO)** is the maximum acceptable time to restore service. These business objectives drive backup frequency, replication, and DR design.

### Q64. How do fault domains contribute to high availability?

**Answer:** Fault domains isolate hardware failures within an availability domain. Distributing redundant instances across fault domains ensures that a hardware or maintenance failure in one fault domain does not take down all replicas, which is essential in single-AD regions.

### Q65. How do you design a multi-region disaster recovery architecture?

**Answer:** Replicate data to a second region (for example, using database replication or Object Storage cross-region replication), maintain infrastructure definitions to rebuild or pre-provision the standby, plan DNS or traffic redirection, define RPO and RTO, and regularly test failover and failback.

### Q66. What role does DNS play in disaster recovery?

**Answer:** DNS, including OCI health-checked traffic management, can redirect users from a failed primary to a recovery site. Effective failover also depends on data replication, application readiness, session handling, and tested procedures rather than DNS changes alone.

### Q67. Why is testing important in a DR plan?

**Answer:** Untested DR plans often fail during real incidents because of missing steps, stale configuration, permission gaps, or unmet RPO and RTO. Regular tests validate procedures, reveal gaps, and build operational confidence.

## 8. Security

### Q68. What is the OCI shared responsibility model?

**Answer:** Oracle secures the cloud infrastructure and physical facilities, while the customer secures their identities, access policies, network configuration, data, and applications. Understanding the boundary prevents gaps where each party assumes the other is responsible.

### Q69. What does OCI Vault provide?

**Answer:** **OCI Vault** manages encryption keys and secrets. Keys protect data through integrated services or cryptographic operations, and secrets store versioned sensitive values. Access, rotation, backup, and deletion protection must be governed carefully.

### Q70. What is the difference between Oracle-managed and customer-managed keys?

**Answer:** **Oracle-managed keys** are handled entirely by the service with less customer control. **Customer-managed keys** in Vault give the customer control over key lifecycle, rotation, and revocation, which can satisfy stricter compliance requirements.

### Q71. What is OCI Cloud Guard?

**Answer:** **Cloud Guard** monitors OCI resources and configurations for security problems and risky activity, using detectors and responders to identify and optionally remediate issues. It helps maintain a consistent security posture across compartments.

### Q72. What are security zones?

**Answer:** **Security Zones** enforce security policies on compartments so that resources cannot be created or changed in ways that violate defined best practices, such as prohibiting public access to certain resources. They add preventive guardrails on top of monitoring.

### Q73. How should network security be layered?

**Answer:** Combine private subnets, NSGs and security lists, controlled gateways, least-privilege IAM, encryption, and monitoring. Defense in depth ensures that a single misconfiguration is less likely to fully expose a workload.

### Q74. How should secrets and API keys be managed?

**Answer:** Store them in OCI Vault, grant access through least-privilege policies and resource or instance principals, rotate them, monitor use, and revoke them when unnecessary. Never place secrets in images, source control, logs, or unprotected configuration.

### Q75. What is the secure response to a suspected credential leak?

**Answer:** Revoke or rotate the credential immediately, determine its permissions and usage from audit logs, contain affected resources, remove the exposure source, replace dependent configuration securely, and add preventive controls. Do not wait for confirmed misuse.

## 9. Observability and Governance

### Q76. What does OCI Monitoring provide?

**Answer:** **Monitoring** collects metrics from OCI resources and custom sources, supports metric queries, and triggers **alarms** that notify operators through Notifications. Alarms should use actionable thresholds and be tested end to end.

### Q77. What does OCI Logging provide?

**Answer:** **Logging** centralizes audit, service, and custom logs. Log groups organize logs, search supports investigation, and connectors can route logs to other services. Retention and access should meet security and compliance requirements.

### Q78. What is the OCI Audit service?

**Answer:** The **Audit** service records API calls made against OCI resources, capturing who did what and when. Audit logs support security investigation, compliance, and change tracking. They are essential when diagnosing configuration or access changes.

### Q79. What is the Notifications service?

**Answer:** **Notifications** delivers messages from sources such as alarms and events to subscriptions like email or functions through topics. It is the delivery mechanism that turns monitoring and automation signals into operator or system actions.

### Q80. What is the OCI Events service?

**Answer:** **Events** matches resource state changes against rules and routes them to targets such as Functions, Notifications, or Streaming. It enables event-driven automation without continuous polling.

### Q81. How is cost tracked and controlled in OCI?

**Answer:** Use compartments and tags for cost attribution, **budgets** with alerts to detect overspending, cost analysis and usage reports for visibility, and right-sizing and scheduled scaling to reduce waste. Governance policies keep spending aligned with ownership.

### Q82. What is the difference between free-form tags and defined tags?

**Answer:** **Free-form tags** are simple key-value labels without a schema. **Defined tags** use a controlled namespace and key definitions, supporting consistent governance, reporting, and policy enforcement, including tag defaults that apply required metadata automatically.

### Q83. What are service limits and quotas?

**Answer:** **Service limits** cap the resources a tenancy can consume for a service, and **quotas** let administrators set compartment-level restrictions. They prevent runaway consumption and support cost and capacity governance. Limit increases can be requested when justified.

## 10. Scenario-Based Interview Questions

### Q84. An application in a private subnet must download OS patches from the internet. What do you configure?

**Answer:** Add a **NAT gateway** and a route rule sending internet-bound traffic from the private subnet to it, keeping instances unreachable from the internet. For OCI service traffic such as Object Storage, use a **service gateway** instead to avoid the public path.

### Q85. Instances in a public subnet cannot reach the internet. What do you check?

**Answer:** Verify the **internet gateway** exists and is attached, the route table sends `0.0.0.0/0` to it, security lists or NSGs allow the traffic, the instance has a public IP, and OS-level firewall rules permit it. Check each layer methodically.

### Q86. You need high availability in a region with only one availability domain. How do you design it?

**Answer:** Distribute instances across the availability domain’s **fault domains**, use an instance pool and a load balancer, replicate data on resilient storage, and use a **second region** for disaster recovery. Fault domains provide the intra-region resilience in single-AD regions.

### Q87. A team needs to manage only compute in one project without touching other resources. How do you grant access?

**Answer:** Place the project resources in a dedicated **compartment**, create a group for the team, and write a policy allowing that group to `manage instance-family` (and any required supporting resources) only in that compartment. This enforces least privilege.

### Q88. Two VCNs need to communicate but have overlapping CIDRs. What do you do?

**Answer:** Peering requires **non-overlapping CIDRs**, so re-address one VCN or design new subnets with distinct ranges. Then configure a local or remote peering connection with appropriate route rules and security rules. Overlapping ranges cannot be peered directly.

### Q89. Object Storage costs are high for data that is rarely accessed. How do you reduce cost?

**Answer:** Move infrequently accessed data to the **Archive** tier or use lifecycle policies to transition objects automatically, accepting the retrieval delay. Review retention and delete unneeded data. Keep frequently accessed data in Standard for low-latency access.

### Q90. You must give a partner temporary upload access to one bucket without an OCI account. What do you use?

**Answer:** Create a **pre-authenticated request** scoped to that bucket with write permission and a short expiry, and share it securely. Treat the URL as a secret and revoke it when no longer needed. This avoids creating and managing external user accounts.

### Q91. A workload on a compute instance needs to call the Object Storage API securely. How do you configure it?

**Answer:** Add the instance to a **dynamic group** and write a policy granting the required Object Storage permissions in the appropriate compartment. The application uses the **instance principal** to authenticate, avoiding stored API keys.

### Q92. Application latency spikes when traffic increases. How do you scale while controlling cost?

**Answer:** Make the tier stateless, place instances behind a load balancer across fault or availability domains, and configure **autoscaling** based on metrics or schedule. Autoscaling adds capacity during demand and removes it afterward to limit cost.

### Q93. You need a shared file system mounted by many instances. Which service and considerations apply?

**Answer:** Use **File Storage**, which provides an NFS-based shared file system accessible by multiple instances. Consider mount targets, network security rules for NFS, performance needs, and access controls. Block Volume is not suitable for concurrent multi-instance sharing in this way.

### Q94. How would you design a secure, highly available three-tier web application on OCI?

**Answer:** Use a regional VCN with public and private subnets, a load balancer in front of stateless web and application instances spread across fault or availability domains, a managed database with standby protection and tested backups, NSGs and least-privilege IAM, Vault-managed keys and secrets, monitoring and alarms, and a documented DR plan to a second region.

### Q95. How do you explain an OCI architecture decision in an interview?

**Answer:** State the requirement (availability, performance, security, or cost), the options considered, the trade-offs, the chosen design, and how it meets RPO, RTO, and governance needs. Show that you balance resilience and cost rather than defaulting to the most complex option.

## 12. Practical Interview Checklist

Before an OCI Architect Associate interview, be prepared to explain:

- Regions, availability domains, fault domains, and single-AD design
- Tenancy, compartments, OCIDs, and organizing resources
- IAM policy syntax, verbs, groups, dynamic groups, and least privilege
- Instance principals and federation
- VCNs, subnets, gateways, route tables, security lists, and NSGs
- VPN, FastConnect, and local and remote VCN peering
- Load balancers, backend sets, and health checks
- Compute shapes, custom images, instance pools, and autoscaling
- Block, Object, File, and Archive storage and when to use each
- Volume backups, clones, volume groups, and pre-authenticated requests
- Autonomous, Base, and Exadata database options
- HA versus DR, RPO and RTO, and multi-region design
- Vault, Cloud Guard, security zones, and the shared responsibility model
- Monitoring, Logging, Audit, Notifications, budgets, tags, and quotas
- At least one end-to-end architecture explained with trade-offs

> **Final interview tip:** In architecture questions, begin with requirements and constraints, justify each service choice against availability, security, and cost, and confirm the design meets RPO, RTO, and least-privilege goals.
