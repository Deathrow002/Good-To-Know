## 1. Messaging & Eventing

### 1.1 IBM MQ

**Overview:**

IBM MQ (formerly MQSeries) is IBM's flagship enterprise message queuing middleware. It provides reliable, once-and-only-once message delivery between applications regardless of transport, platform, or network reliability.

**Key concepts:**

- **Queue Manager (QM):** Core server process that manages queues and channels. All MQ objects belong to a queue manager.
- **Queue:** Named storage for messages. Types: Local, Remote, Alias, Model, Dead-Letter Queue (DLQ).
- **Channel:** Communication link between queue managers or between a client and queue manager. Types: Sender/Receiver, Server/Requester, Client-Connection, Cluster.
- **Message:** Data unit containing a payload and an MQMD (Message Descriptor) header with metadata (message ID, correlation ID, persistence, priority, expiry).
- **Put / Get:** Core verbs for sending and receiving messages. `MQPUT` writes to a queue; `MQGET` reads from it.
- **Publish/Subscribe:** Topics and subscriptions allow pub/sub messaging alongside point-to-point queuing.
- **Clustering:** Queue managers can form a cluster to provide load balancing and high availability.
- **MQ Network / Multi-hop:** Messages can traverse multiple queue managers to reach their destination.

**Delivery guarantees:**

| Guarantee | How |
|---|---|
| **Persistent messages** | Written to disk; survive queue manager restart |
| **Transactional put/get** | Wrapped in a unit of work (local or XA global transaction) |
| **Exactly-once delivery** | Persistent + transactional = no duplicates, no loss |

**High availability options:**

- **Multi-Instance Queue Manager:** Active/standby with shared storage (NFS/SAN).
- **Uniform Cluster:** Multiple active queue managers with automatic rebalancing of client connections.
- **IBM MQ on OpenShift (via CP4I):** Container-native HA using the MQ Operator.

**Common use cases:**

- Decoupling microservices and legacy enterprise systems.
- Guaranteed delivery for financial transactions, order processing, and audit trails.
- Bridging heterogeneous platforms (mainframe, Linux, Windows, cloud).
- Request/reply patterns and workflow coordination.

### 1.3 IBM Event Streams

**Overview:**

**Key features:**

- Geo-replication (Mirroring) for cross-region disaster recovery.
- IBM Cloud IAM integration for authentication and authorization.
- Kafka REST Proxy for HTTP-based access.
- Built-in metrics, dashboards, and alerting via IBM Instana or Prometheus.

**Deployment options:**

| Option | Description |
|---|---|
| **IBM Cloud (SaaS)** | Fully managed; Lite, Standard, Enterprise plans |
| **CP4I on OpenShift** | Self-managed via Event Streams Operator |

**Differences from vanilla Kafka:**

- Hardened security (TLS by default, mutual TLS, IAM RBAC).
- Automated broker management, topic management UI.
- IBM support and SLA backed.

## 2. API Management

### 2.1 IBM API Connect

**Overview:**

IBM API Connect is a full lifecycle API management platform that allows organizations to create, secure, manage, socialize, and analyze APIs. It supports REST, SOAP, GraphQL, and event-driven (AsyncAPI) APIs.

**Core components:**

| Component | Role |
|---|---|
| **API Manager** | Web UI for designing, publishing, and managing APIs and products |
| **API Gateway (DataPower)** | Runtime enforcement of security, rate limiting, transformation, and routing |
| **Developer Portal** | Self-service portal for API consumers to discover, subscribe, and test APIs |
| **Analytics** | API usage metrics, dashboards, and alerting |
| **Cloud Manager** | Multi-cloud/multi-org administration |

**Key concepts:**

- **API Definition:** OpenAPI (Swagger) or WSDL-based specification of an API's resources, operations, and parameters.
- **Product:** Packaging of one or more APIs with a plan (rate limits, quotas) for publication to the Developer Portal.
- **Plan:** Defines rate limits and quotas for a subscription to a product.
- **Application:** Registered consumer app that holds credentials (client ID + secret) to call APIs.
- **Subscription:** Binding of an application to a plan, granting access to the product's APIs.
- **Catalog:** Environment-level grouping of products (e.g., Sandbox, Production).
- **Space:** Sub-partition of a catalog for delegation of publish/manage permissions.
- **Gateway Policy Assembly:** Pipeline of policies (invoke, set-variable, gatewayscript, map, rate-limit, etc.) applied to API requests and responses.

**Security capabilities:**

- OAuth 2.0 / OpenID Connect flows (authorization code, client credentials, PKCE).
- API key and basic auth enforcement.
- TLS mutual authentication.
- JWT validation and generation.
- IP filtering and threat protection.

**Deployment options:**

| Option | Description |
|---|---|
| **SaaS (IBM Cloud)** | Fully managed API Connect on IBM Cloud |
| **CP4I on OpenShift** | Self-managed via API Connect Operator |
| **On-Premises** | VMware or bare-metal deployment |

### 2.2 IBM DataPower Gateway

**Overview:**

IBM DataPower Gateway is a purpose-built, high-performance hardware appliance and virtual/container software gateway that enforces security, transformation, and routing for APIs, web services, and B2B traffic at the network edge.

**Key capabilities:**

## 3. Integration Platform

### 3.1 IBM App Connect Enterprise (ACE)

**Overview:**

IBM App Connect Enterprise (ACE) is IBM's strategic enterprise integration platform, the evolution of IBM Integration Bus (IIB) and WebSphere Message Broker. It provides a runtime and tooling for building message flows that connect applications, services, and data across protocols and formats.

**Core concepts:**

| Concept | Description |
|---|---|
| **Integration Server** | Runtime process that hosts and executes message flows |
| **Integration Node** | Management entity (in legacy IIB mode) that governs one or more integration servers |
| **Message Flow** | Visual pipeline of connected nodes that processes messages from input to output |
| **Flow Node** | Individual processing step in a flow (e.g., Kafka Consumer, HTTP Request, MQ Input, Mapping, Filter, Compute) |
| **Message** | Data being processed; consists of a logical tree (parsed), bit-stream (unparsed), and metadata (environment, local env, exception list) |
| **Parser / Domain** | Component that parses message bit-streams into logical trees (XMLNSC, JSON, BLOB, MRM, DFDL, etc.) |
| **Mapping Node** | Graphical or ESQL-based data transformation; supports ESQL, Java, XPath, XSLT |
| **ESQL** | Extended SQL scripting language used in Compute and Filter nodes for message transformation and routing |
| **Policy** | Externalised configuration for connections (MQ, Kafka, JDBC, HTTP) decoupled from flows |
| **BAR File** | Broker Archive — deployable artefact containing compiled flows and configuration |

**Connectivity:** HTTP/HTTPS, IBM MQ, Apache Kafka, JMS, JDBC, REST, SOAP, FTP/SFTP, Email, SAP, Salesforce, and more via built-in nodes and connectors.

**Message formats supported:** XML, JSON, CSV, fixed-width, HL7, SWIFT, EDI, binary (DFDL-based).

**Deployment options:**

| Option | Description |
|---|---|
| **Container (CP4I)** | ACE Operator on OpenShift; Integration Server as a pod |
| **On-Premises** | Installed on Linux/Windows servers |
| **IBM Cloud** | Managed App Connect flows (SaaS) |

---

### 3.2 IBM App Connect (SaaS / Designer)

**Overview:**

IBM App Connect is the cloud-native, low-code/no-code integration service on IBM Cloud (formerly IBM App Connect Professional / CastIron). It provides a drag-and-drop flow designer for connecting SaaS applications and APIs without writing code.

## 4. B2B & File Transfer

### 4.1 IBM Sterling B2B Integrator

**Overview:**

IBM Sterling B2B Integrator is a comprehensive B2B integration platform that manages EDI transactions, business process automation, and multi-enterprise data exchange.

**Key capabilities:**

- EDI translation: X12, EDIFACT, RosettaNet, HL7, SWIFT, tradacoms.
- Business Process Modelling Language (BPML) for orchestrating multi-step B2B processes.
- Partner onboarding and trading partner management.
- Protocols: AS2, AS4, SFTP, FTP, HTTP/S, OFTP, ebMS.
- Integration with ERP systems (SAP, Oracle) and IBM MQ/MFT.
- Compliance and audit trails for regulatory requirements.

---

### 4.2 IBM Sterling File Gateway

**Overview:**

IBM Sterling File Gateway is built on Sterling B2B Integrator and provides a self-service partner portal and workflow engine specifically for managed file transfer between trading partners.

**Key concepts:**

- **Channels:** Named file routing definitions matching file patterns to routing rules.
- **Routing Rules:** Business logic directing inbound files from producers to consumers.
- **Producer / Consumer:** Trading partner roles for sending and receiving files.
- **Mailbox:** Storage location for partner files within the platform.

---

### 4.3 IBM Sterling Connect:Direct

**Overview:**

IBM Sterling Connect:Direct (formerly NDM — Network Data Mover) is a point-to-point file transfer product designed for high-volume, high-reliability transfers between servers, mainframes, and cloud environments.

**Key features:**

- Checkpoint restart: resumes interrupted transfers from the last checkpoint without retransmitting the whole file.
- Compression and encryption (TLS/AES).
- Process scripting language for defining transfer jobs and conditional logic.
- Platform support: z/OS, UNIX, Linux, Windows, IBM i.
- Used by banks, insurers, and government agencies for large-scale batch file movement.

---

### 4.4 IBM Managed File Transfer (MFT)

**Overview:**

IBM MQ Managed File Transfer (MFT) is a file transfer capability built on IBM MQ transport, providing auditable, reliable, and secure file movement within and between organizations.

**Key features:**

- Files are transferred as segmented MQ messages; reliability and ordering are inherited from MQ.
- Central audit database records all transfer requests, progress, and outcomes.
- Trigger-based transfers from MQ messages, schedules, or file-system events.
- PGP encryption and digital signing support.
- Included in IBM MQ Advanced licensing.

---

### 4.5 IBM Sterling Secure Proxy

**Overview:**

IBM Sterling Secure Proxy is a DMZ gateway that acts as a security perimeter for B2B communications. It terminates external connections in the DMZ and re-initiates them to internal Sterling B2B Integrator servers — keeping internal systems hidden from external networks.

**Protocols supported:** AS2, SFTP, FTP/S, Connect:Direct, HTTP/S.

## 5. Cloud Pak for Integration (CP4I)

### 5.1 Overview

**IBM Cloud Pak for Integration (CP4I)** is IBM's unified, container-based integration platform deployed on Red Hat OpenShift. It bundles the core IBM integration capabilities under a single license and operator-based deployment model.

**Bundled capabilities:**

| Capability | Product |
|---|---|
| **Messaging** | IBM MQ |
| **Event Streaming** | IBM Event Streams (Kafka) |
| **API Management** | IBM API Connect |
| **API Gateway** | IBM DataPower Gateway |
| **Enterprise Integration** | IBM App Connect Enterprise (ACE) |
| **Cloud-Native Flows** | IBM App Connect Designer |
| **High-Speed Transfer** | IBM Aspera on Cloud |
| **Operations Dashboard** | Distributed tracing (OpenTracing / Jaeger) across all capabilities |

**Architecture:**

```
Red Hat OpenShift Cluster
├── Platform Navigator (unified UI / license entitlement hub)
├── IBM MQ Operator          → Queue Manager pods
├── Event Streams Operator   → Kafka cluster pods
├── API Connect Operator     → API Manager, Gateway, Portal, Analytics
├── ACE Operator             → Integration Server pods (BAR deployments)
├── App Connect Designer     → Flow runtime pods
├── DataPower Operator       → Gateway pods
├── Aspera HSTS Operator     → Transfer server pods
└── Operations Dashboard     → Tracing + observability
```

**Key benefits:**

- **Single license:** All capabilities are included; no per-product licensing complexity.
- **Operator-driven lifecycle:** Deploy, upgrade, and scale each capability independently via Kubernetes Operators.
- **GitOps-ready:** All configuration is Kubernetes custom resources (CRs), deployable via ArgoCD or Flux.
- **Unified observability:** Operations Dashboard provides end-to-end distributed tracing across messaging, APIs, and integration flows.
- **Multi-cloud:** Runs on any OpenShift cluster — on-premises, AWS ROSA, Azure ARO, IBM Cloud ROKS, Google Anthos.

---

### 5.2 Platform Navigator

The **Platform Navigator** is the unified web console and entitlement manager for CP4I. It provides:

- Single sign-on to all CP4I capability consoles.
- Instance creation and management for each capability (e.g., deploy a new queue manager, a new ACE integration server, a new API Connect cluster).
- License usage tracking across all deployed capabilities.

## 6. Process & Event Automation

### 6.1 IBM Event Automation

**Overview:**

IBM Event Automation is IBM's platform for processing, correlating, and acting on event streams in real time. It is built on Apache Kafka and Apache Flink and delivered as part of CP4I.

**Components:**

| Component | Purpose |
|---|---|
| **Event Streams** | Kafka-based event backbone (source of events) |
| **Event Processing** | No-code / low-code Flink-based stream processing for filtering, correlating, and detecting event patterns |
| **Event Endpoint Management** | Manage, describe, and govern event sources as self-service API endpoints (AsyncAPI-based) |

**Key use cases:**

- Detect complex event patterns (e.g., fraud: multiple failed logins followed by a large transaction).
- Enrich events by joining streams with reference data.
- Filter and route events to downstream systems.
- Expose Kafka topics as governed, discoverable event endpoints with self-service subscriptions.

---

### 6.2 IBM Business Automation Workflow (BAW)

**Overview:**

IBM Business Automation Workflow is a BPM and case management platform that combines process design, execution, monitoring, and optimization. It integrates with IBM MQ, ACE, and external REST/SOAP services to orchestrate business processes.

**Key features:**

- BPMN 2.0 process design and execution.
- Case management for unstructured / dynamic work.
- Process Portal for human task management.
- Integration with IBM Operational Decision Manager for rule-driven decision points.
- REST, SOAP, IBM MQ, and Java connector support.

---

### 6.3 IBM Operational Decision Manager (ODM)

**Overview:**

IBM ODM is a business rules management system (BRMS) that externalises business logic from application code into manageable, versioned rules that business analysts can maintain without developer involvement.

**Components:**

| Component | Purpose |
|---|---|
| **Decision Center** | Web-based rules authoring and governance for business users |
| **Rule Execution Server (RES)** | Runtime engine that executes rule sets against input data |
| **Decision Server Insights** | Event-driven rules for complex event processing |

**Integration:** Embedded in BAW process flows, called from ACE flows, REST API invocation from microservices.

## 8. High-Speed File Transfer

### 8.1 IBM Aspera

**Overview:**

IBM Aspera is a high-speed data transfer technology designed to move large files and data sets over WANs at speeds close to the theoretical line rate, regardless of file size, transfer distance, or network conditions.

**Core technology — FASP Protocol:**

IBM Aspera uses its proprietary **FASP (Fast, Adaptive, and Secure Protocol)** instead of TCP for transport. FASP:

- Uses UDP for data transport with a custom reliability and congestion control layer.
- Fully utilizes available bandwidth regardless of latency or packet loss (where TCP degrades severely at high RTT).
- Provides encryption (AES-256) in transit at full line speed.
- Achieves near line-rate throughput on intercontinental transfers where TCP would achieve a fraction.

**Products in the Aspera family:**

| Product | Use Case |
|---|---|
| **Aspera on Cloud** | SaaS collaboration and transfer portal |
| **Aspera HSTS (High-Speed Transfer Server)** | On-premises/cloud transfer server |
| **Aspera Connect** | Browser plugin / desktop client for end-user transfers |
| **Aspera Desktop Client** | Standalone GUI client |
| **Aspera Cargo** | Background transfer daemon for automated transfers |
| **Aspera Drive** | Cloud-drive-like sync client |
| **Aspera Shares** | Collaboration and file sharing portal |
| **Aspera Faspex** | Email-like file delivery portal |
| **Aspera Orchestrator** | Workflow automation for multi-step transfer pipelines |

**Common use cases:**

- Media and entertainment: uploading/downloading large video files between studios, post-production houses, and cloud storage.
- Life sciences and genomics: transferring terabyte-scale genome data sets globally.
- Financial services: end-of-day batch file delivery between global offices.
- Government and intelligence: large secure data transfers across classified networks.

## 10. Integration Decision Guide

```
Need guaranteed, transactional messaging between enterprise apps?
  └─► IBM MQ

Need high-throughput event streaming / Kafka?
  └─► IBM Event Streams

Need to manage, secure, and publish APIs externally?
  └─► IBM API Connect + DataPower Gateway

Need to integrate heterogeneous systems with complex transformations?
  └─► IBM App Connect Enterprise (ACE)

Need to connect SaaS apps with low-code flows?
  └─► IBM App Connect (Designer / SaaS)

Need B2B EDI and multi-enterprise data exchange?
  └─► IBM Sterling B2B Integrator

Need reliable, auditable file transfers at scale?
  └─► IBM MFT (within org) or IBM Sterling Connect:Direct (point-to-point)

Need to move terabytes of data across WANs at line rate?
  └─► IBM Aspera

Need all of the above on OpenShift with a single license?
  └─► IBM Cloud Pak for Integration (CP4I)
```
