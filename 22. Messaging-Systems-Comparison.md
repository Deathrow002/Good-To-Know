---

## 2. Architecture

### Apache Kafka
- **Broker-based:** A cluster of brokers stores messages in partitioned, replicated logs (topics).
- **Topics & Partitions:** Messages are written to topic partitions sequentially. Each partition is an ordered, immutable log.
- **Consumer Groups:** Consumers in a group share partition assignment; each partition consumed by exactly one member.
- **ZooKeeper / KRaft:** Older versions use ZooKeeper for metadata; KRaft mode (Kafka 3.x+) replaces ZooKeeper.
- **Retention:** Messages persist on disk for a configurable retention period regardless of consumption.

### RabbitMQ
- **Broker-based:** A central broker (or cluster) manages queues and exchanges.
- **Exchanges & Queues:** Producers publish to exchanges; exchanges route to queues based on bindings and routing keys.
- **Exchange Types:** Direct, Fanout, Topic, Headers — enabling highly flexible routing patterns.
- **Push Model:** Broker pushes messages to consumers; messages are deleted after acknowledgement.
- **Plugins:** Extensible via plugins (e.g., MQTT, delayed messages, federation, shovel).

### ActiveMQ
- **Broker-based:** Single or clustered broker managing queues and topics.
- **JMS Native:** Built around the Java Message Service (JMS) specification — Queues (P2P) and Topics (Pub/Sub).
- **OpenWire Protocol:** Default native protocol; also supports AMQP, STOMP, MQTT, WebSockets.
- **ActiveMQ Artemis:** Next-generation broker with non-blocking I/O, higher throughput; basis for HornetQ.
- **Message Selectors:** Consumers can filter messages server-side via SQL-92 expressions.

### IBM MQ
- **Queue Manager:** Central component managing queues and channels; highly reliable.
- **Channels:** Communication links between queue managers or between clients and queue managers.
- **Persistent by Default:** Messages are guaranteed persistent with transactional support.
- **MQ Network:** Queue managers interconnect to form a messaging network for enterprise topologies.
- **Mainframe Support:** Deep integration with IBM z/OS, CICS, IMS — unique in the industry.

## 3. Messaging Model

| Aspect | Kafka | RabbitMQ | ActiveMQ | IBM MQ |
|---|---|---|---|---|
| **Message Retention** | Configurable (log retention) — consumers can replay | Deleted after ACK | Deleted after ACK | Deleted after delivery (persistent on disk until then) |
| **Replay / Rewind** | Yes — consumers control offsets | No | No | No |
| **Message Ordering** | Guaranteed per partition | Per-queue (single consumer) | Per-queue (single consumer) | Per-queue |
| **Routing** | Simple (topic/partition) | Sophisticated (exchanges, bindings, routing keys) | Moderate (queues, topics, selectors) | Moderate (queues, pub/sub topics) |
| **Consumer Model** | Pull (consumers poll) | Push (broker pushes) | Push (broker pushes) | Push |
| **At-Least-Once** | Yes | Yes | Yes | Yes |
| **Exactly-Once** | Yes (idempotent producer + transactional) | Plugin-based (limited) | Limited | Yes (with XA transactions) |

## 4. Performance & Scalability

| Metric | Kafka | RabbitMQ | ActiveMQ | IBM MQ |
|---|---|---|---|---|
| **Throughput** | Very high (millions of msgs/sec) | High (tens of thousands/sec) | Moderate | Moderate |
| **Latency** | Low (ms range, slightly higher than RabbitMQ) | Very low (sub-ms possible) | Low to moderate | Moderate |
| **Horizontal Scaling** | Native (add brokers/partitions) | Clustering (mirrored queues / quorum queues) | Clustering (shared storage or network-of-brokers) | Queue manager clusters (limited horizontal scale) |
| **Partition/Shard Support** | Yes — core design principle | Quorum queues provide replication, not sharding | Limited | No native sharding |
| **Best For Volume** | Extremely high-volume event streams | High-volume task queues, flexible routing | Moderate-volume JMS apps | Enterprise guaranteed delivery |

## 5. Durability & Reliability

| Feature | Kafka | RabbitMQ | ActiveMQ | IBM MQ |
|---|---|---|---|---|
| **Persistence** | Yes — always (log on disk) | Configurable per message/queue | Configurable per message/queue | Yes — persistent by default |
| **Replication** | Yes — configurable replication factor | Yes — quorum queues or mirrored queues | Yes — via network of brokers or shared storage | Yes — multi-instance high availability |
| **Transactions** | Yes (idempotent + transactional APIs) | Yes (AMQP transactions, publisher confirms) | Yes (JMS transactions, XA) | Yes (full XA, two-phase commit) |
| **Dead Letter Queue** | Manual (redirect to another topic) | Native DLQ support | Native DLQ support | Native DLQ support |
| **Fault Tolerance** | High — distributed replication | High — quorum queues | Moderate | Very High — IBM enterprise HA |

## 9. Use Cases

### Apache Kafka — Best For:
- **Event streaming at scale:** Real-time data pipelines processing millions of events per second.
- **Event sourcing:** Storing domain events as the system of record with replay capability.
- **Change Data Capture (CDC):** Streaming database changes via Debezium to downstream consumers.
- **Log aggregation:** Collecting and centralizing logs, metrics, and telemetry from many services.
- **Real-time analytics:** Feeding analytics dashboards, ML models, and alerting systems.
- **Microservice event bus:** Decoupling microservices with durable, replayable event streams.

### RabbitMQ — Best For:
- **Task / work queues:** Distributing background jobs (email sending, image processing) across workers.
- **Flexible message routing:** Complex routing topologies using exchanges, bindings, and routing keys.
- **Request/Reply (RPC):** Implementing synchronous-style RPC over async messaging.
- **Microservice communication:** Lightweight service-to-service messaging with AMQP.
- **IoT messaging:** Using MQTT plugin for lightweight IoT device communication.
- **Delayed / scheduled tasks:** Using delayed message exchange plugin.

### ActiveMQ — Best For:
- **JMS-native Java applications:** Legacy enterprise Java apps using JMS API.
- **Enterprise application integration (EAI):** Integrating heterogeneous systems via JMS.
- **Complex message routing with Apache Camel:** Often used alongside Camel for EIP patterns.
- **Moderate throughput messaging:** Enterprise apps not requiring Kafka-scale throughput.
- **Transaction-heavy systems:** Applications requiring full JMS transaction support.

### IBM MQ — Best For:
- **Mainframe / z/OS integration:** Only serious option for connecting mainframe applications.
- **Banking & financial services:** Where guaranteed delivery, auditability, and compliance are critical.
- **Enterprise integration with legacy IBM systems:** CICS, IMS, WebSphere integrations.
- **Regulated industries:** Healthcare, finance, government requiring enterprise-grade reliability and certifications.
- **Exactly-once delivery:** Full XA transaction support across heterogeneous systems.

## 10. When NOT to Use Each

| Broker | Avoid When... |
|---|---|
| **Kafka** | You need complex routing, RPC patterns, or very low message counts; short-lived/transient messages; simple task queues |
| **RabbitMQ** | You need message replay/rewind; extremely high throughput (millions/sec); event sourcing |
| **ActiveMQ** | Starting a greenfield project (prefer Artemis or RabbitMQ); need very high throughput |
| **IBM MQ** | You have no IBM ecosystem; cost is a concern; you need open-source flexibility |

---

## 11. Quick Decision Guide

```
Is message replay / event sourcing required?
  YES → Kafka

Is mainframe / IBM z/OS integration required?
  YES → IBM MQ

Is guaranteed delivery in regulated/financial context critical?
  YES (with IBM ecosystem) → IBM MQ
  YES (without IBM) → RabbitMQ or Kafka

Do you need complex routing (topic-based, header-based, fanout)?
  YES → RabbitMQ

Is the application JMS-based (legacy Java enterprise)?
  YES → ActiveMQ (or ActiveMQ Artemis)

Do you need extremely high throughput (millions of events/sec)?
  YES → Kafka

Do you need simple task queues / work distribution?
  YES → RabbitMQ
```
