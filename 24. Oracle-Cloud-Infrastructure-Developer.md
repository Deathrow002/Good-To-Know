# Oracle Cloud Infrastructure 2023 Certified Developer Professional — Interview Questions and Answers

This guide contains original interview questions and concise answers aligned with **Oracle Cloud Infrastructure 2023 Certified Developer Professional (1Z0-1084-23)** skills. It covers cloud-native architecture, containers, Oracle Kubernetes Engine, Functions, API Gateway, events and messaging, OCI DevOps, observability, security, and troubleshooting.

> **Note:** This is an interview-preparation guide, not an exam dump. The 2023 exam is retired or superseded in some regions, and OCI services continue to evolve. Verify current service behavior, limits, and certification objectives in official Oracle documentation.

---

## 1. Cloud-Native and Microservices Fundamentals

### Q1. What is a cloud-native application?

**Answer:** A cloud-native application is designed to use elastic infrastructure, automation, managed services, immutable deployment, observability, and failure-aware architecture. It is usually composed of independently deployable components and can scale or recover without relying on a single server.

### Q2. What are the main characteristics of microservices?

**Answer:** Microservices are small, independently deployable services organized around business capabilities. Each service owns its implementation and usually its data, communicates through well-defined APIs or events, and can be developed, deployed, and scaled independently.

### Q3. What are the benefits and costs of microservices?

**Answer:** Benefits include team autonomy, independent scaling, fault isolation, and technology flexibility. Costs include distributed transactions, network latency, data consistency challenges, more complex testing, observability, security, and deployment operations. A modular monolith can be better when those costs are not justified.

### Q4. What is the difference between horizontal and vertical scaling?

**Answer:** **Vertical scaling** adds CPU or memory to an existing instance. **Horizontal scaling** adds more instances and distributes traffic among them. Cloud-native stateless services generally favor horizontal scaling because it improves elasticity and fault tolerance.

### Q5. Why should application services be stateless?

**Answer:** Stateless instances can be replaced or scaled without moving local session data. Durable state should be stored in an appropriate database, object store, cache, or messaging service. Statelessness simplifies load balancing, rolling deployment, and failure recovery.

### Q6. What is the twelve-factor configuration principle?

**Answer:** Configuration that varies between deployments should be externalized from application code. On OCI, configuration can come from environment variables, Kubernetes ConfigMaps, Vault secrets, or a managed configuration mechanism. Secrets must not be committed to source control.

### Q7. What is an idempotent operation?

**Answer:** An idempotent operation produces the same intended state when repeated. It is important because clients, queues, functions, and deployment systems may retry requests. Common techniques include idempotency keys, unique constraints, deduplication records, and conditional writes.

### Q8. What is eventual consistency?

**Answer:** Eventual consistency means replicas or services can temporarily hold different values but converge when updates finish propagating. Applications must handle stale reads, duplicate events, reordered events, and compensating actions when strong distributed transactions are unavailable.

### Q9. What is the circuit-breaker pattern?

**Answer:** A circuit breaker stops calls to a dependency after repeated failures, fails fast for a period, and later probes for recovery. It prevents resource exhaustion and cascading failure. It should be combined with bounded retries, timeouts, fallback behavior, and monitoring.

### Q10. What is the strangler pattern?

**Answer:** The strangler pattern incrementally replaces a legacy application by routing selected capabilities to new services while the old system continues to operate. It reduces migration risk compared with a single large rewrite.

> **Interview tip:** Do not claim that every application should use microservices. Explain the operational trade-offs.

## 2. Containers, Docker, and OCI Container Registry

### Q11. What is the difference between a container and a virtual machine?

**Answer:** A virtual machine includes a guest operating system and virtualized hardware. A container packages an application and its dependencies but shares the host kernel. Containers are usually smaller and start faster, while VMs provide a stronger infrastructure isolation boundary.

### Q12. What is a container image?

**Answer:** A container image is an immutable, layered package containing an application, runtime, libraries, and metadata. A container is a runtime instance of that image with an additional writable layer.

### Q13. Why use a multi-stage Docker build?

**Answer:** A multi-stage build compiles or packages the application in one stage and copies only required runtime artifacts into a smaller final image. This reduces image size, attack surface, and accidental inclusion of source code or build credentials.

```dockerfile
FROM eclipse-temurin:21-jdk AS build
WORKDIR /src
COPY . .
RUN ./mvnw -q -DskipTests package

FROM eclipse-temurin:21-jre
RUN useradd --system --uid 10001 appuser
COPY --from=build /src/target/app.jar /app/app.jar
USER 10001
ENTRYPOINT ["java", "-jar", "/app/app.jar"]
```

### Q14. Why should a container run as a non-root user?

**Answer:** It limits the impact of a process compromise and supports least privilege. The image and Kubernetes security context should define a non-root user, avoid unnecessary Linux capabilities, and use a read-only root file system where possible.

### Q15. What is Oracle Cloud Infrastructure Registry?

**Answer:** **OCI Container Registry**, commonly called **OCIR**, is an OCI-managed registry for storing and distributing Open Container Initiative-compatible images. Repositories can be public or private and are controlled through OCI IAM.

### Q16. How does a developer authenticate Docker to OCIR?

**Answer:** A user commonly logs in with the regional registry endpoint, a username that includes the tenancy namespace and user identity, and an OCI authentication token as the password. Automation should use an approved non-human identity or secret mechanism rather than a personal credential.

### Q17. What does an OCIR image name look like?

**Answer:** It follows this general form:

```text
<region-key>.ocir.io/<tenancy-namespace>/<repository>:<tag>
```

The region key, tenancy namespace, repository path, and tag must match the target registry and image.

### Q18. Why should deployments use image digests instead of mutable tags?

**Answer:** A digest identifies exact image content, while a tag such as `latest` can be moved to another image. Digest pinning improves repeatability, auditability, and rollback safety. Release pipelines can still apply meaningful immutable version tags.

### Q19. How should container images be secured?

**Answer:** Use trusted minimal base images, patch dependencies, scan for vulnerabilities, sign or attest images where supported, exclude secrets, run as non-root, minimize packages and capabilities, and rebuild regularly. Admission controls can reject unapproved images.

### Q20. Why can a container work locally but fail in OCI?

**Answer:** Common causes include incorrect CPU architecture, missing configuration or secrets, reliance on local files, a wrong listening interface or port, insufficient resources, private registry authentication, network policies, or differences in identity and DNS.

## 3. Oracle Kubernetes Engine

### Q21. What is Oracle Kubernetes Engine?

**Answer:** **Oracle Kubernetes Engine (OKE)** is an OCI-managed Kubernetes service. OCI manages the Kubernetes control plane, while worker-node responsibility depends on whether managed nodes, virtual nodes, or other supported options are used.

### Q22. What are the main Kubernetes control-plane components?

**Answer:** The API server exposes the Kubernetes API, `etcd` stores cluster state, the scheduler assigns pods to nodes, and controller managers reconcile desired and actual state. In OKE, OCI operates the managed control plane.

### Q23. What is the difference between a Pod and a Deployment?

**Answer:** A Pod is the smallest deployable Kubernetes unit and contains one or more tightly coupled containers. A Deployment manages ReplicaSets and provides declarative replica management, rolling updates, and rollback for stateless workloads.

### Q24. What is the difference between a Deployment and a StatefulSet?

**Answer:** A Deployment manages interchangeable pods. A StatefulSet provides stable pod identities, ordered operations, and stable storage claims, making it suitable for workloads that require persistent identity or ordered coordination.

### Q25. What Kubernetes Service types are commonly used in OKE?

**Answer:**

| Service type | Purpose |
|---|---|
| ClusterIP | Internal virtual IP accessible within the cluster |
| NodePort | Exposes a port on each worker node |
| LoadBalancer | Provisions or integrates with an OCI load balancer |
| ExternalName | Maps a service name to an external DNS name |

### Q26. What is an Ingress?

**Answer:** An Ingress defines HTTP or HTTPS routing rules to cluster services. It requires an Ingress controller or supported OCI-native implementation. It centralizes host and path routing, TLS termination, and exposure of multiple services.

### Q27. What are readiness, liveness, and startup probes?

**Answer:** A **readiness probe** controls whether a pod receives traffic. A **liveness probe** detects a stuck application and can trigger restart. A **startup probe** protects slow-starting applications from premature liveness failures. Probe endpoints should reflect their distinct purposes.

### Q28. What is the difference between resource requests and limits?

**Answer:** Requests influence scheduling and represent reserved capacity. Limits cap permitted resource use. A CPU limit can throttle a container, while exceeding a memory limit can cause an out-of-memory termination. Values should be based on measured workload behavior.

### Q29. How does a rolling update work?

**Answer:** A Deployment gradually creates pods using the new template and removes old pods according to `maxSurge` and `maxUnavailable`. Readiness probes, termination handling, compatible database changes, and sufficient spare capacity are required for a safe rollout.

### Q30. How do you roll back a failed Kubernetes deployment?

**Answer:** Stop or undo the rollout to a known ReplicaSet, verify readiness and application behavior, and diagnose the failed version. Rollback is reliable only if configuration, external dependencies, and database schema remain backward-compatible.

### Q31. What is a ConfigMap, and when should it not be used?

**Answer:** A ConfigMap stores non-sensitive configuration for pods. It should not store passwords, tokens, private keys, or other secrets. Sensitive material belongs in Kubernetes Secrets with proper encryption and controls or, preferably, an external managed secret store.

### Q32. How can an OKE workload access an OCI service securely?

**Answer:** Use the supported workload or resource identity mechanism and grant narrowly scoped IAM policies. Avoid embedding user API keys in images or Kubernetes manifests. Also restrict network access and audit service calls.

### Q33. What is a Kubernetes NetworkPolicy?

**Answer:** A NetworkPolicy specifies allowed pod ingress or egress based on pod, namespace, and IP selectors. It is effective only when the cluster networking implementation enforces it. A default-deny baseline with explicit allowances reduces lateral movement.

### Q34. What is the Horizontal Pod Autoscaler?

**Answer:** The **HPA** adjusts pod replica count from observed metrics such as CPU, memory, or custom metrics. It requires usable metrics and realistic resource requests. It does not replace node capacity planning or application-level load testing.

### Q35. How does cluster autoscaling differ from HPA?

**Answer:** HPA changes the number of pods. A cluster autoscaler changes worker-node capacity when pods cannot be scheduled or nodes are underused. They solve different layers of the scaling problem and are often used together.

### Q36. A pod is in `Pending`. What do you check?

**Answer:** Inspect pod events and scheduler messages. Check resource requests, node capacity, taints and tolerations, affinity rules, persistent-volume binding, node selectors, quotas, and autoscaler health.

### Q37. A pod is in `CrashLoopBackOff`. What do you check?

**Answer:** Inspect current and previous container logs, exit code, events, command and arguments, environment, mounted secrets, dependencies, probes, and resource limits. Fix the application or configuration rather than repeatedly deleting the pod.

### Q38. A pod cannot pull an image from OCIR. What are likely causes?

**Answer:** Check the image path and tag, repository existence, region endpoint, IAM permission, image-pull secret or workload identity, node egress and DNS, and image architecture. Kubernetes events usually reveal authentication or not-found errors.

### Q39. How should persistent storage be requested in Kubernetes?

**Answer:** A workload creates a PersistentVolumeClaim referencing a StorageClass and requested capacity and access mode. The CSI driver dynamically provisions a suitable OCI volume where supported. Backup and availability requirements must be designed separately.

### Q40. How do you design OKE for high availability?

**Answer:** Distribute worker nodes and pod replicas across fault or availability domains, use topology spread or anti-affinity, configure disruption budgets, use resilient load balancing and storage, avoid single-replica dependencies, and test node and zone failure.

> **Interview tip:** Kubernetes restarts containers; it does not automatically make the application correct, stateful, or highly available.

## 4. OCI Functions and Serverless Development

### Q41. What is OCI Functions?

**Answer:** OCI Functions is a managed functions-as-a-service platform based on the Fn Project. Developers deploy containerized functions and OCI handles invocation, scaling, and runtime infrastructure. It is suitable for event-driven and short-lived workloads within documented limits.

### Q42. What is the relationship between a function and a Functions application?

**Answer:** A function is deployable code and configuration. A Functions application is a logical grouping that provides common configuration and network context for its functions. The application is associated with subnets used for required network access.

### Q43. When should OCI Functions not be used?

**Answer:** Avoid it for processes that exceed execution limits, require long-lived connections or local durable state, need highly specialized host control, or have steady workloads better served by containers or compute. Service limits and cold-start sensitivity should be evaluated.

### Q44. What is a cold start?

**Answer:** A cold start occurs when the platform must initialize an execution environment before processing a request. Package size, runtime initialization, dependency loading, and network setup can add latency. Keep functions small and initialize reusable clients outside the handler when appropriate.

### Q45. How should function configuration and secrets be managed?

**Answer:** Store non-sensitive settings as application or function configuration. Store credentials and keys in OCI Vault and let the function retrieve them using a resource principal with least-privilege policies. Never bake secrets into the function image.

### Q46. What is a resource principal?

**Answer:** A resource principal lets an OCI resource authenticate to OCI APIs without a user API key. Dynamic-group matching rules identify the resource, and IAM policies grant allowed actions. This supports short-lived, automatically managed credentials.

### Q47. How can a function be invoked?

**Answer:** It can be invoked directly through the Functions API or CLI, synchronously through API Gateway, or asynchronously through integrations such as Events and other services. The invocation method affects authentication, timeout, retry, and response behavior.

### Q48. How should function retries be handled?

**Answer:** Assume an invocation can occur more than once. Make side effects idempotent, use unique event identifiers, persist completion state where necessary, bound downstream retries, and route unrecoverable work to a durable failure-handling path.

### Q49. Why should a function avoid relying on local disk?

**Answer:** Function instances are ephemeral and can be replaced between invocations. Temporary disk can be used only within documented limits during an invocation. Durable data belongs in Object Storage, a database, or another persistent service.

### Q50. How do you troubleshoot a failed function invocation?

**Answer:** Correlate the invocation identifier with logs, verify image deployment and handler configuration, inspect IAM policies and dynamic-group membership, check application subnet routes and security rules, validate payload format, and examine timeout or memory usage.

## 5. API Gateway

### Q51. What is OCI API Gateway?

**Answer:** OCI API Gateway is a managed service for publishing HTTP APIs with routing, authentication, request or response policies, rate controls, logging, and back-end integration. Back ends can include Functions, HTTP services, and other supported targets.

### Q52. What is an API deployment?

**Answer:** An API deployment defines the path prefix, routes, methods, back ends, and route policies published on a gateway. Treat the specification as version-controlled configuration and validate it before deployment.

### Q53. How can API Gateway authenticate clients?

**Answer:** Depending on the design, it can use IAM-backed authorization, JSON Web Token validation with an identity provider, custom authorization functions, or other supported mechanisms. Authentication proves identity; route authorization must still restrict permitted operations.

### Q54. What is the difference between authentication and authorization?

**Answer:** Authentication establishes who the caller is. Authorization determines what that caller may do. A valid token should not automatically grant access to every route or resource.

### Q55. Why use rate limiting?

**Answer:** Rate limiting protects back ends from accidental overload, abuse, and noisy clients. Limits should align with customer plans and downstream capacity. Clients should receive a clear response and use exponential backoff with jitter where retry is appropriate.

### Q56. How should API versions be managed?

**Answer:** Prefer backward-compatible evolution. For breaking changes, publish a new explicit version, support an overlap period, document migration, measure old-version usage, and retire it through a communicated policy.

### Q57. What is CORS?

**Answer:** Cross-Origin Resource Sharing is a browser security mechanism controlling whether scripts from one origin can access resources at another. Configure only required origins, methods, and headers; a permissive wildcard policy can expose APIs unnecessarily.

### Q58. API Gateway returns an error while the back end works directly. What do you inspect?

**Answer:** Inspect the deployment route and method, authentication policy, request transformation, back-end URL or function identifier, gateway network path, timeout, response mapping, execution logs, and correlation identifiers.

## 6. Events, Streaming, Queue, and Service Integration

### Q59. What is OCI Events?

**Answer:** OCI Events routes matching state-change events from OCI resources to supported targets. Rules use event patterns, and targets can include Functions, Notifications, and Streaming. It enables loosely coupled automation without continuously polling services.

### Q60. Should event consumers assume exactly-once delivery?

**Answer:** No. Consumers should tolerate duplicate delivery, retry, and possible ordering differences. Use idempotency, deduplication, and durable state to make repeated processing safe.

### Q61. What is OCI Streaming?

**Answer:** OCI Streaming is a managed, partitioned event-streaming service for high-throughput ordered records within each partition. Producers publish messages and consumers track offsets to process or replay the stream.

### Q62. How is ordering achieved in OCI Streaming?

**Answer:** Ordering is guaranteed within a partition, not across an entire multi-partition stream. Use a stable message key so related events map to the same partition. This preserves entity order but can create hot partitions if keys are poorly distributed.

### Q63. What is a consumer group?

**Answer:** A consumer group coordinates consumers so partitions are assigned among group members. It enables parallel processing while each partition is handled by one member at a time within the group. Rebalancing can occur when membership changes.

### Q64. What is the purpose of a cursor in Streaming?

**Answer:** A cursor identifies where a consumer begins reading, such as a particular offset, time, latest position, or group-managed position. Cursor type determines replay behavior and should match the application’s recovery needs.

### Q65. What is OCI Queue?

**Answer:** OCI Queue is a managed service for asynchronous message delivery between producers and consumers. Consumers retrieve messages, process them, and delete them after success. Visibility timeouts and dead-letter handling support reliable work processing.

### Q66. Streaming or Queue: how do you choose?

**Answer:**

| Requirement | Prefer |
|---|---|
| Replayable ordered event history | Streaming |
| Partitioned high-throughput event log | Streaming |
| Distributing independent work items | Queue |
| Per-message acknowledgement and visibility timeout | Queue |
| Multiple independent views of retained events | Streaming |

### Q67. What is a visibility timeout?

**Answer:** After a queue consumer receives a message, the message is hidden for a period. If it is not deleted before that period expires, it becomes visible for retry. The consumer may need to extend visibility for long processing while remaining idempotent.

### Q68. What is a dead-letter queue?

**Answer:** A dead-letter queue stores messages that cannot be processed successfully after the configured delivery attempts. Operators should alert on it, preserve diagnostic context, correct the cause, and use a controlled replay process.

### Q69. What is OCI Service Connector Hub?

**Answer:** Service Connector Hub moves data between supported OCI source and target services, optionally using a function for transformation. It is commonly used to route logs to Object Storage, Streaming, Monitoring, or other destinations without custom polling infrastructure.

### Q70. How do you prevent a poison message from blocking processing?

**Answer:** Limit retries, capture the error and message identifier, move the message to a dead-letter destination, alert an owner, and continue processing other work. Fix and replay it through an audited procedure.

## 7. OCI DevOps and Delivery Automation

### Q71. What does OCI DevOps provide?

**Answer:** OCI DevOps provides managed source-code repositories, build pipelines, artifact integration, deployment pipelines, environments, approvals, and deployment strategies for OCI workloads.

### Q72. What is the difference between a build pipeline and a deployment pipeline?

**Answer:** A build pipeline checks out source, runs build and test steps, creates artifacts, and publishes them. A deployment pipeline promotes a chosen immutable artifact through target environments using deployment stages and controls.

### Q73. What is a build specification?

**Answer:** A build specification is a version-controlled YAML document describing build steps, commands, environment variables, output artifacts, and related configuration. It makes the build repeatable and reviewable.

### Q74. Why should CI/CD pipelines produce immutable artifacts?

**Answer:** The same tested artifact should be promoted through environments. Rebuilding for production can introduce dependency or source differences. Images should be referenced by immutable tags or digests and binaries by fixed versions or checksums.

### Q75. What is blue-green deployment?

**Answer:** Blue-green deployment maintains the current and new environments simultaneously. Traffic switches to the new environment after validation, allowing rapid rollback by switching back. It requires extra capacity and backward-compatible external state.

### Q76. What is canary deployment?

**Answer:** Canary deployment sends a small portion of traffic to a new version and increases exposure after health checks pass. It reduces blast radius but requires representative routing, useful metrics, automated analysis, and rollback thresholds.

### Q77. What is a rolling deployment?

**Answer:** A rolling deployment replaces instances incrementally while keeping part of the old version available. It uses less duplicate capacity than blue-green but temporarily runs mixed versions and can make rollback slower.

### Q78. How should database schema changes be deployed with an application?

**Answer:** Use backward-compatible expand-and-contract changes. Add compatible schema first, deploy code that can work with both forms, migrate data, stop old usage, and remove obsolete schema later. Avoid coupling an irreversible schema change to an immediate application rollout.

### Q79. How should secrets be made available to a build?

**Answer:** Retrieve them at runtime from an approved secret store using the build resource’s identity and narrowly scoped policy. Mask output, avoid writing secrets into artifacts or caches, and rotate any credential that might have been exposed.

### Q80. What should stop an automated production deployment?

**Answer:** Failed tests, vulnerable or unapproved artifacts, missing approvals, unhealthy target capacity, breached error or latency thresholds, failed smoke tests, or loss of observability should stop or roll back deployment according to policy.

### Q81. A build works locally but fails in OCI DevOps. What do you check?

**Answer:** Compare runtime and tool versions, CPU architecture, build context, network access to dependencies, IAM permissions, secret availability, file-name case, environment variables, resource limits, and commands in the build specification.

### Q82. How should rollback be designed?

**Answer:** Keep the last known-good immutable artifact and deployment configuration, automate traffic or version reversal, maintain backward-compatible state, define measurable rollback triggers, and regularly test the procedure.

## 8. Observability and Troubleshooting

### Q83. What are the three primary observability signals?

**Answer:** **Metrics** show numeric trends, **logs** provide discrete diagnostic records, and **traces** follow a request through distributed components. Effective troubleshooting correlates all three using consistent timestamps and identifiers.

### Q84. What does OCI Logging provide?

**Answer:** OCI Logging centralizes supported service logs and custom application logs. Log groups organize logs, search supports investigation, and connectors can route records to other services. Retention and access should match security and compliance requirements.

### Q85. What is a correlation ID?

**Answer:** A correlation ID is propagated through an incoming request, downstream calls, messages, logs, and traces. It allows operators to reconstruct one transaction across API Gateway, functions, containers, and managed services.

### Q86. What makes an application log useful?

**Answer:** Use structured records with timestamp, severity, service and version, environment, correlation ID, event name, and relevant non-sensitive context. Avoid passwords, tokens, personal data, unbounded payloads, and ambiguous free-form messages.

### Q87. What does OCI Monitoring provide?

**Answer:** OCI Monitoring stores service and custom metrics, supports metric queries, and triggers alarms. Alarms can notify operators through OCI Notifications. Thresholds should be actionable and tested rather than copied without workload context.

### Q88. What is distributed tracing?

**Answer:** Distributed tracing records spans representing work across service boundaries. It helps locate latency, errors, and dependency bottlenecks. Sampling must balance visibility, cost, and overhead while preserving enough traces for incident analysis.

### Q89. How do you investigate an increase in API latency?

**Answer:** Confirm the affected routes and time window, compare request rate and error rate, inspect gateway and application metrics, follow traces, identify slow dependencies, check recent deployments, resource saturation, retries, and network changes, then validate recovery against a baseline.

### Q90. How should alarms be designed for cloud-native applications?

**Answer:** Alert on customer impact and service-level indicators such as sustained errors, latency, queue age, failed invocations, or unavailable replicas. Use meaningful windows, severity, ownership, runbook links, and deduplication to avoid alert fatigue.

## 9. IAM, Security, and Secrets

### Q91. How does OCI IAM authorize API calls?

**Answer:** A principal authenticates, OCI policies evaluate whether that principal or its group can perform a verb on a resource type in a location, and service-specific conditions can narrow access. Compartments provide administrative scope, not a network boundary.

### Q92. What are the common OCI policy verbs?

**Answer:** From least to most permissive, they are generally `inspect`, `read`, `use`, and `manage`. Their exact permissions depend on the resource type. Prefer the least verb and scope that supports the required operation.

### Q93. What is the difference between a user group and a dynamic group?

**Answer:** A user group contains human or federated IAM users. A dynamic group identifies OCI resources using matching rules. Policies grant permissions to either group, but workloads should generally use resource principals rather than user credentials.

### Q94. What does OCI Vault provide?

**Answer:** OCI Vault provides managed keys and secret storage. Keys can protect data through integrated services or cryptographic APIs, while secrets hold versioned sensitive values. Access, rotation, backup, and deletion controls must be explicitly governed.

### Q95. How should API keys and authentication tokens be treated?

**Answer:** Treat them as secrets: minimize creation, scope their owner’s permissions, store them securely, rotate them, monitor use, and revoke them when no longer needed. Never place them in source control, images, logs, or public pipeline variables.

### Q96. What is least privilege in a developer workflow?

**Answer:** Developers, pipelines, functions, and cluster workloads receive only the actions needed on specific resources and compartments. Separate build from deploy permission, production from non-production, and human from workload identities.

### Q97. How do network security and IAM complement each other?

**Answer:** IAM controls which authenticated principal can call an OCI API. Network controls restrict which paths and endpoints can be reached. Strong architecture applies both and also enforces application-level authorization and encryption.

### Q98. What is the secure response to a leaked credential?

**Answer:** Revoke or rotate it immediately, identify its permissions and usage, inspect audit and application logs, contain affected resources, remove the exposure source, replace dependent configuration safely, and document preventive controls. Do not wait for proof of misuse.

## 10. Scenario-Based Interview Questions

### Q99. A function receives duplicate events and creates duplicate orders. How do you fix it?

**Answer:** Use a stable event or business idempotency key, enforce uniqueness in durable storage, record processing state transactionally, and return the prior result for repeats. Keep retries because removing them would trade duplication for lost work.

### Q100. An OKE deployment is healthy, but users receive intermittent errors. What do you check?

**Answer:** Check readiness behavior, service endpoints, pod and node distribution, per-pod logs, resource throttling, connection pools, ingress or load-balancer health, mixed application versions, downstream dependencies, and whether terminating pods still receive traffic.

### Q101. Messages are accumulating in a queue. What is your response?

**Answer:** Compare arrival and processing rates, inspect message age and failure reasons, verify consumer health and scaling, check downstream latency, visibility timeout, poison messages, and dead-letter counts. Scale only after identifying whether consumers or dependencies are the bottleneck.

### Q102. A new image has a critical vulnerability after deployment. What do you do?

**Answer:** Assess exploitability and exposure, stop promotion, roll back or rebuild from a patched base image, redeploy the immutable replacement, rotate potentially exposed secrets, scan related images, and add a pipeline policy to prevent recurrence.

### Q103. A function times out while calling a private database. What do you inspect?

**Answer:** Check application subnet selection, route tables, NSGs or security lists, DNS, database listener and credentials, connection timeout, pool initialization, function duration and memory, and logs on both sides. Reusing a safely initialized client can reduce repeated connection cost.

### Q104. An API is being overloaded by one client. How do you protect it?

**Answer:** Authenticate the caller, enforce client-specific rate or usage limits, return a suitable throttling response, protect downstream concurrency, monitor abuse, and coordinate a quota change if the traffic is legitimate.

### Q105. A canary release increases latency but not errors. Should it proceed?

**Answer:** Compare latency with the defined service objective and canary guardrails. If the increase is material, stop promotion or roll back, inspect traces and resource metrics, and diagnose before exposing more traffic. Error rate alone is not sufficient.

### Q106. A pod needs to read an Object Storage bucket. How do you grant access?

**Answer:** Use the supported OKE workload identity or resource-principal approach, create a narrowly scoped IAM policy for the required bucket operations, avoid static user keys, restrict egress where appropriate, and verify access through Audit logs.

### Q107. A stream consumer processes records out of business order. What might be wrong?

**Answer:** Related records may use different partition keys or the business requires ordering across partitions, which Streaming does not provide. Select a stable entity key, handle sequence numbers in the consumer, or redesign the workflow to tolerate reordering.

### Q108. How would you design a secure OCI event-processing application?

**Answer:** Use private endpoints where supported, resource principals, least-privilege policies, Vault secrets, encrypted services, schema validation, idempotent consumers, bounded retries, dead-letter handling, structured logs, metrics, tracing, immutable delivery, and tested recovery procedures.

### Q109. How do you explain a cloud-native production incident in an interview?

**Answer:** State the customer impact, timeline, detection signal, evidence, containment, root cause, restoration, and measurable prevention. Explain trade-offs and ownership without blaming individuals, and distinguish the trigger from underlying systemic weaknesses.

### Q110. What would a production-ready OCI delivery path look like?

**Answer:** Commit reviewed source, run reproducible tests and security scans, build one immutable artifact, publish it to an approved registry or repository, promote it through environments, use automated health gates and approvals, deploy progressively, observe service-level indicators, and support automated rollback.

- Secure, minimal container images and OCIR authentication
- OKE workloads, services, probes, storage, networking, and autoscaling
- Kubernetes rollout, rollback, and common pod failures
- Functions applications, invocation, cold starts, and resource principals
- API Gateway routes, authentication, rate limiting, and troubleshooting
- Events, Streaming partitions and cursors, Queue visibility, and dead letters
- OCI DevOps builds, immutable artifacts, approvals, and deployment strategies
- Metrics, logs, traces, correlation IDs, alarms, and incident diagnosis
- IAM policies, dynamic groups, Vault, secret rotation, and least privilege
- Backward-compatible database changes and safe progressive delivery
- At least two incidents described with impact, evidence, recovery, and prevention

> **Final interview tip:** In scenario answers, start with customer impact and evidence, protect data and credentials, use managed identities and immutable artifacts, and finish with verification and a preventive control.
