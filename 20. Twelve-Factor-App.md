# The Twelve-Factor App — Questions & Answers

A methodology for building **software-as-a-service (SaaS)** and cloud-native applications that are portable, scalable, and easy to deploy. Originally formulated by engineers at Heroku, the Twelve-Factor App is a set of best practices that keeps applications resilient and maintainable in modern cloud environments.

---

## 1. What is the Twelve-Factor App methodology, and why does it matter?

**Answer:**

The **Twelve-Factor App** is a methodology for building modern, scalable, maintainable software-as-a-service applications. It defines twelve principles that guide how an application should be built and deployed so it can run reliably across different environments (development, staging, production) and scale horizontally in the cloud.

---

### a. Why it matters

1.  **Portability:** Minimizes divergence between environments, so the app runs the same way everywhere.
2.  **Scalability:** Designed to scale out (add more instances) rather than scale up (bigger machines).
3.  **Continuous Deployment:** Reduces friction between writing code and running it in production.
4.  **Cloud-Native Fit:** Maps directly onto container platforms (Docker, Kubernetes) and PaaS offerings.

---

### b. The twelve factors at a glance

| # | Factor | Core Idea |
|---|---|---|
| 1 | **Codebase** | One codebase tracked in version control, many deploys. |
| 2 | **Dependencies** | Explicitly declare and isolate dependencies. |
| 3 | **Config** | Store configuration in the environment. |
| 4 | **Backing Services** | Treat backing services as attached resources. |
| 5 | **Build, Release, Run** | Strictly separate build and run stages. |
| 6 | **Processes** | Execute the app as stateless processes. |
| 7 | **Port Binding** | Export services via port binding. |
| 8 | **Concurrency** | Scale out via the process model. |
| 9 | **Disposability** | Maximize robustness with fast startup and graceful shutdown. |
| 10 | **Dev/Prod Parity** | Keep development, staging, and production as similar as possible. |
| 11 | **Logs** | Treat logs as event streams. |
| 12 | **Admin Processes** | Run admin/management tasks as one-off processes. |

## 2. Factor I — Codebase: One codebase tracked in revision control, many deploys

**Answer:**

There should be a **one-to-one correlation between the codebase and the app**. A single codebase is tracked in a version control system (like Git), and many deploys (production, staging, developer laptops) run from that same codebase.

---

### a. Key rules
- **One codebase per app.** If there are multiple codebases, it's not one app — it's a distributed system of many apps.
- **Shared code** belongs in libraries pulled in via the dependency manager, not copied between repos.
- A **deploy** is a running instance of the app. Different deploys may run different versions (e.g., a feature branch on staging), but they all share the same codebase.

### b. Spring Boot example
A single Git repository contains your Spring Boot app. The same JAR is deployed to dev, QA, and prod — only the environment configuration differs.

---

## 3. Factor II — Dependencies: Explicitly declare and isolate dependencies

**Answer:**

A twelve-factor app **never relies on the implicit existence of system-wide packages**. It declares *all* dependencies completely and explicitly via a dependency declaration manifest, and isolates them so they don't leak in from the surrounding system.

---

### a. How it's applied
- **Declaration:** List every dependency with a specific version in a manifest.
- **Isolation:** Use a tool that guarantees no implicit dependencies from the host system "leak in."

### b. Java/Spring example
```xml
<!-- pom.xml — explicit declaration with managed versions -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
```
- **Maven/Gradle** act as the dependency declaration and isolation tool.
- The embedded server (Tomcat) is bundled into the build, so the app does **not** depend on a pre-installed application server.

---

## 4. Factor III — Config: Store config in the environment

**Answer:**

**Configuration that varies between deploys** (credentials, hostnames, resource handles, feature flags) must be stored in the **environment**, not hardcoded in the source. A strict separation of config from code is required.

---

### a. The litmus test
> Could you make the codebase open source at any moment **without compromising any credentials**?

If yes, config is correctly externalized.

### b. What counts as config
- Database/connection URLs and credentials
- Credentials to backing services (caches, queues, third-party APIs)
- Per-deploy values like the canonical hostname

> **Note:** Internal application config that does *not* vary between deploys (e.g., framework routing) is **not** considered config under this factor.

### c. Spring Boot example
```yaml
# application.yml uses environment variables
spring:
  datasource:
    url: ${DB_URL}
    username: ${DB_USER}
    password: ${DB_PASSWORD}
```
Spring's `@Value`, `Environment`, and `@ConfigurationProperties` read these values from environment variables or externalized property sources — never from committed secrets.

---

## 5. Factor IV — Backing Services: Treat backing services as attached resources

**Answer:**

A **backing service** is any service the app consumes over the network — databases, message brokers (Kafka, RabbitMQ), caches (Redis), SMTP servers, or third-party APIs. The twelve-factor app makes **no distinction between local and third-party services**; both are attached resources accessed via a URL/handle stored in config.

---

### a. Key idea
- Resources can be **attached and detached** to deploys at will, without code changes.
- Swapping a local MySQL for an Amazon RDS instance should only require a **config change** (the connection URL), not a code change.

### b. Example
```yaml
spring:
  data:
    redis:
      url: ${REDIS_URL}   # local Redis or managed cloud Redis — same code
```

---

## 6. Factor V — Build, Release, Run: Strictly separate build and run stages

**Answer:**

A codebase is transformed into a deploy through **three strictly separated stages**:

1.  **Build:** Convert code into an executable bundle (compile, fetch dependencies, package the artifact — e.g., a JAR).
2.  **Release:** Combine the build with the deploy's **config** to create a release. Each release has a unique ID and is immutable.
3.  **Run:** Execute the release in the runtime environment (start the processes).

---

### a. Key rules
- **No code changes at runtime** — you cannot modify the build once it's running; changes require a new build/release.
- **Every release is immutable and uniquely identified**, enabling easy rollback to a previous release.

### b. CI/CD mapping
- **Build:** `mvn package` → produces `app.jar`.
- **Release:** `app.jar` + environment config → versioned Docker image / release.
- **Run:** `java -jar app.jar` or `docker run` in the target environment.

---

## 7. Factor VI — Processes: Execute the app as one or more stateless processes

**Answer:**

Twelve-factor processes are **stateless** and **share-nothing**. Any data that needs to persist must be stored in a stateful **backing service** (typically a database). Memory or local disk may be used only as a brief, single-transaction cache.

---

### a. Why stateless matters
- Enables **horizontal scaling** — any instance can handle any request.
- An instance can crash and be replaced without losing user data.

### b. Common violation
**Sticky sessions** — caching user session data in the app's memory and relying on future requests routing to the same instance — **violate** this factor. Instead, store session state in a shared store like Redis (e.g., Spring Session).

---

## 8. Factor VII — Port Binding: Export services via port binding

**Answer:**

The app is **completely self-contained** and exports its services by **binding to a port**. It does not rely on runtime injection of an external web server (like a separately installed Tomcat or Apache) to function.

---

### a. How it's applied
- The app includes its own server (an **embedded server**) and listens on a port.
- One app can become the **backing service** for another, simply by providing a URL.

### b. Spring Boot example
Spring Boot embeds Tomcat/Netty and binds to a port directly:
```yaml
server:
  port: 8080
```
Running `java -jar app.jar` is enough — no external servlet container required.

---

## 9. Factor VIII — Concurrency: Scale out via the process model

**Answer:**

The app should **scale out horizontally** by running multiple processes, rather than scaling up a single large process. Different types of work are assigned to different **process types** (e.g., web processes for HTTP, worker processes for background jobs).

---

### a. Key ideas
- Processes are **first-class citizens**, modeled after the Unix process model for running services.
- The app should **never daemonize itself or write PID files**; instead, rely on the platform's process manager (systemd, Kubernetes, the cloud platform) for management.
- Scaling is achieved by adding more processes/instances.

### b. Example
- **Web process type:** handles incoming REST requests.
- **Worker process type:** consumes Kafka messages or processes scheduled jobs.
Each type scales independently based on its own load.

---

## 10. Factor IX — Disposability: Maximize robustness with fast startup and graceful shutdown

**Answer:**

Processes are **disposable** — they can be **started or stopped at a moment's notice**. This enables fast elastic scaling, rapid deployment of code/config changes, and robust production behavior.

---

### a. Requirements
1.  **Fast startup:** Processes should boot in seconds, making scaling and recovery quick.
2.  **Graceful shutdown:** On receiving a `SIGTERM`, the process should stop accepting new requests, finish in-flight requests, return jobs to the queue, then exit.
3.  **Robust against sudden death:** Should hardware fail, the app must handle abrupt termination — backing services and queues should allow work to be reprocessed.

### b. Spring Boot support
```yaml
server:
  shutdown: graceful
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```
This lets Spring Boot drain active requests before terminating.

---

## 11. Factor X — Dev/Prod Parity: Keep development, staging, and production as similar as possible

**Answer:**

The twelve-factor app is designed for **continuous deployment** by keeping the gap between development and production **small**. It minimizes three types of gaps:

| Gap | Twelve-Factor Goal |
|---|---|
| **Time gap** | Hours between writing and deploying code (not weeks). |
| **Personnel gap** | Developers who write code are involved in deploying and monitoring it. |
| **Tools gap** | Same backing services across all environments. |

---

### a. Key principle
**Resist the temptation to use different backing services** between dev and prod (e.g., an in-memory H2 database locally but PostgreSQL in production). Lightweight local substitutes introduce subtle incompatibilities that pass tests but break in production.

### b. Best practice
Use the **same type and version** of every backing service in all environments — Docker and tools like Testcontainers make this practical.

---

## 12. Factor XI — Logs: Treat logs as event streams

**Answer:**

A twelve-factor app **never concerns itself with routing or storage of its output stream**. It should not write to or manage log files. Instead, each running process writes its **event stream, unbuffered, to `stdout`**.

---

### a. How it works
- The app simply logs to standard output.
- The **execution environment** captures the stream and routes it to its final destination (a log indexing system like the ELK Stack, Splunk, or a cloud logging service).
- This decouples the app from log management concerns.

### b. Spring Boot example
By default, Spring Boot logs to the console. In containers, you avoid file appenders and let the platform (Docker/Kubernetes) collect `stdout`:
```
docker logs <container>   # platform aggregates the stream
```

---

## 13. Factor XII — Admin Processes: Run admin/management tasks as one-off processes

**Answer:**

**Administrative or management tasks** (database migrations, running a console/REPL, one-time scripts) should be run as **one-off processes** in an environment **identical** to the app's regular long-running processes.

---

### a. Key rules
- Admin code ships **with the application code** (same codebase) to avoid synchronization issues.
- One-off processes run against the **same release** (same build + config) as the running app.
- They use the **same dependency isolation** techniques.

### b. Examples
- **Database migrations:** `Flyway` or `Liquibase` run on app startup or as a dedicated migration job.
- **One-off scripts:** Executed via the same container image, e.g., `docker run myapp:1.4 java -jar app.jar --migrate`.

## 14. How do the Twelve Factors map to a Spring Boot / cloud-native stack?

**Answer:**

| Factor | Spring Boot / Cloud-Native Implementation |
|---|---|
| **Codebase** | Single Git repo, one CI/CD pipeline. |
| **Dependencies** | Maven/Gradle with managed versions; embedded server. |
| **Config** | `application.yml` + environment variables, `@ConfigurationProperties`, Spring Cloud Config. |
| **Backing Services** | DataSource, Kafka, RabbitMQ, Redis injected via config URLs. |
| **Build, Release, Run** | `mvn package` → Docker image → `docker run` / Kubernetes Deployment. |
| **Processes** | Stateless controllers; session state in Redis (Spring Session). |
| **Port Binding** | Embedded Tomcat/Netty on `server.port`. |
| **Concurrency** | Multiple pods/replicas; separate web and worker deployments. |
| **Disposability** | `server.shutdown: graceful`; liveness/readiness probes. |
| **Dev/Prod Parity** | Testcontainers; same DB version everywhere. |
| **Logs** | Log to `stdout`; aggregated by the platform (ELK/Loki). |
| **Admin Processes** | Flyway/Liquibase migrations; one-off jobs from the same image. |

## 15. What are common anti-patterns that violate the Twelve-Factor methodology?

**Answer:**

| Anti-Pattern | Factor Violated | Fix |
|---|---|---|
| Hardcoding credentials in `application.properties` | III — Config | Externalize to environment variables / secrets manager. |
| Storing session state in app memory (sticky sessions) | VI — Processes | Use a shared store (Redis / Spring Session). |
| Using H2 in dev but PostgreSQL in prod | X — Dev/Prod Parity | Use the same DB type/version everywhere. |
| Writing logs to local files managed by the app | XI — Logs | Write to `stdout`; let the platform route logs. |
| Editing code directly on the running server | V — Build/Release/Run | Rebuild and redeploy a new immutable release. |
| Relying on a pre-installed external Tomcat | VII — Port Binding | Use the embedded server packaged in the JAR. |
| Sharing one codebase across multiple apps | I — Codebase | Split into separate apps or extract shared libraries. |
| Manual, multi-week release cycles | X — Dev/Prod Parity | Adopt CI/CD for continuous deployment. |

## 16. Are there extensions beyond the original Twelve Factors?

**Answer:**

Yes. As cloud-native architectures and microservices matured, practitioners proposed additional factors (popularized by Kevin Hoffman in *Beyond the Twelve-Factor App*):

1.  **API First:** Design and agree on the API contract before implementation.
2.  **Telemetry:** Treat monitoring, metrics, and tracing as first-class concerns (e.g., Micrometer, OpenTelemetry, Spring Boot Actuator).
3.  **Authentication & Authorization:** Bake security into the app from the start (e.g., OAuth2 / JWT with Spring Security).

These complement — but do not replace — the original twelve principles, addressing operational and security needs that are critical for modern microservices and Kubernetes-based deployments.
