## Architecture & Internals

### Q1. Explain the Docker architecture and how its components interact.

**Answer:**

Docker uses a **client-server architecture** consisting of:

- **Docker Client (`docker`)** — the CLI tool that users interact with. It sends commands to the Docker daemon via REST API over a Unix socket or TCP.
- **Docker Daemon (`dockerd`)** — the long-running background process that manages images, containers, networks, and volumes.
- **containerd** — an industry-standard container runtime that `dockerd` delegates lifecycle management to (pulling images, managing storage, executing containers).
- **runc** — the low-level OCI-compliant container runtime that actually creates and starts containers by interacting with the Linux kernel (namespaces, cgroups).
- **Docker Registry** — stores and distributes Docker images (e.g., Docker Hub, ECR, GCR).

**Request flow:**
```
docker run nginx
  → Docker Client
    → dockerd (REST API)
      → containerd
        → containerd-shim
          → runc (creates container)
```

### Q2. What Linux kernel features does Docker rely on, and what is each responsible for?

**Answer:**

Docker is fundamentally a packaging and management layer over native Linux kernel primitives:

| Kernel Feature | Purpose |
|---|---|
| **Namespaces** | Isolate container resources (PID, NET, MNT, UTS, IPC, USER) |
| **cgroups (Control Groups)** | Limit and account for resource usage (CPU, memory, I/O, network) |
| **Union File Systems** | Layer-based image storage (OverlayFS, AUFS, devicemapper) |
| **Seccomp** | Filter system calls available to containers |
| **Capabilities** | Grant fine-grained kernel privileges instead of full root |
| **Netfilter/iptables** | Manage container networking and port forwarding |

**Namespace types:**
- `pid` — isolated process tree; container PID 1 is not host PID 1
- `net` — isolated network stack (interfaces, routes, iptables)
- `mnt` — isolated mount points and filesystem
- `uts` — isolated hostname and domain name
- `ipc` — isolated inter-process communication
- `user` — isolated UID/GID mappings (user namespaces, rootless Docker)

### Q3. How does the Docker image layering system (OverlayFS) work?

**Answer:**

Docker uses **OverlayFS** (overlay2 driver) to implement a union filesystem. Each image consists of read-only layers stacked on top of each other. When a container runs, a thin **read-write layer** is added on top.

**OverlayFS components:**
- **lowerdir** — one or more read-only image layers
- **upperdir** — the container's writable layer
- **workdir** — internal OverlayFS work directory
- **merged** — the unified view presented to the container

**Copy-on-Write (CoW):** When a container modifies a file from a lower layer, OverlayFS copies that file up to `upperdir` first, then modifies it. The original in `lowerdir` is unchanged.

```
Container View (merged):
┌──────────────────────────┐
│  upperdir (RW layer)     │  ← container writes here
├──────────────────────────┤
│  lower layer 3 (RO)      │  ← RUN apt-get install
├──────────────────────────┤
│  lower layer 2 (RO)      │  ← COPY app files
├──────────────────────────┤
│  lower layer 1 (RO)      │  ← base image (ubuntu)
└──────────────────────────┘
```

**Implication:** Layers are shared across images and containers that use the same base, reducing disk usage significantly.

### Q4. What is the difference between `CMD` and `ENTRYPOINT`? How do they interact?

**Answer:**

Both define what runs when a container starts, but they behave differently:

| | `CMD` | `ENTRYPOINT` |
|---|---|---|
| **Purpose** | Default arguments / default command | The fixed executable to run |
| **Overridable?** | Yes, by `docker run <image> <command>` | Only with `--entrypoint` flag |
| **Forms** | Shell form or exec form | Shell form or exec form |

**Interaction:** When both are defined, `CMD` provides **default arguments** to `ENTRYPOINT`.

```dockerfile
# Example: flexible container with defaults
ENTRYPOINT ["python", "app.py"]
CMD ["--port", "8080"]        # default args, overridable

# docker run myapp            → python app.py --port 8080
# docker run myapp --port 9090 → python app.py --port 9090
```

**Shell form vs Exec form:**

```dockerfile
# Shell form — runs via /bin/sh -c (signals NOT forwarded to process)
CMD python app.py

# Exec form — runs directly (signals ARE forwarded, preferred for PID 1)
CMD ["python", "app.py"]
```

Always prefer **exec form** so that the process receives OS signals (SIGTERM) correctly, enabling graceful shutdown.

### Q5. What is a multi-stage build and why is it important in production?

**Answer:**

Multi-stage builds allow you to use multiple `FROM` statements in a single Dockerfile. Each `FROM` starts a new build stage. You can **copy artifacts** from one stage to another, leaving behind build tools and intermediate files.

**Benefits:**
- Dramatically smaller final images (no compiler, dev dependencies, or build cache)
- Better security (fewer installed packages = smaller attack surface)
- Single Dockerfile for the full pipeline

```dockerfile
# Stage 1: Build
FROM golang:1.22 AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o server ./cmd/server

# Stage 2: Production image
FROM gcr.io/distroless/static:nonroot
COPY --from=builder /app/server /server
USER nonroot:nonroot
ENTRYPOINT ["/server"]
```

**Result:** A Go app image that's ~10MB instead of ~800MB (golang base image).

## Networking

### Q6. Describe Docker's networking modes and when to use each.

**Answer:**

| Mode | Description | Use Case |
|---|---|---|
| **bridge** (default) | Virtual private network; containers get their own IP | Isolated multi-container apps on a single host |
| **host** | Container shares host's network stack directly | High-performance, low-latency networking; monitoring agents |
| **none** | No networking; fully isolated | Security-sensitive batch jobs, air-gapped workloads |
| **overlay** | Multi-host virtual network (requires Swarm/k8s) | Distributed services across multiple Docker hosts |
| **macvlan** | Container gets its own MAC address on host network | Legacy apps expecting a physical network presence |
| **ipvlan** | Similar to macvlan but shares MAC; L2/L3 modes | More scalable than macvlan on large networks |

**Key behavior of user-defined bridge networks:**
- Provide **automatic DNS resolution** between containers by container name
- More isolated than the default `bridge` network
- Support `--network-alias` for load-balancing

```bash
# Create and use a custom bridge
docker network create my-net
docker run --network my-net --name api myapp
docker run --network my-net --name db postgres

# "api" container can reach "db" by hostname
```

### Q7. How does Docker DNS work in user-defined networks?

**Answer:**

Docker runs an **embedded DNS server** at `127.0.0.11` within each user-defined network. When a container is connected to such a network, its `/etc/resolv.conf` is configured to use this DNS server.

**Resolution order:**
1. Container name (exact match)
2. Network alias (`--network-alias`)
3. Service name (in Docker Compose)
4. External DNS (host's configured nameservers as upstream)

**Docker Compose example:**

```yaml
services:
  web:
    image: nginx
  api:
    image: myapi
    # can reach "web" by hostname "web"
  db:
    image: postgres
    # can reach by hostname "db"
```

The default `bridge0` network does **not** support DNS resolution by container name — only user-defined networks do. This is a key reason to always create custom networks.

### Q8. Explain how Docker port mapping works internally (iptables and NAT).

**Answer:**

When you run `docker run -p 8080:80 nginx`, Docker:

1. Sets up a **DNAT (Destination NAT) rule** in iptables to forward packets arriving on host port `8080` to the container's IP and port `80`.
2. Creates a **MASQUERADE rule** for outbound traffic (SNAT) so the container can reach external networks.

**iptables chains involved:**
- `DOCKER` chain in `nat` table — contains DNAT rules per published port
- `DOCKER-USER` chain — user-customizable rules that run before Docker's own
- `DOCKER-ISOLATION-STAGE-1/2` — prevent inter-network communication

```bash
# View Docker's iptables rules
sudo iptables -t nat -L -n -v
sudo iptables -L DOCKER -n -v
```

**Important:** Rules in `DOCKER-USER` chain are preserved when the daemon restarts; rules added directly to `DOCKER` chain are not.

## Storage & Volumes

### Q9. What are the differences between bind mounts, volumes, and tmpfs mounts?

**Answer:**

| Feature | Volumes | Bind Mounts | tmpfs |
|---|---|---|---|
| **Managed by** | Docker | User / host OS | Docker (in memory) |
| **Host path** | `/var/lib/docker/volumes/` | Any host path | RAM only, not on disk |
| **Persists after container removal** | Yes (until explicitly removed) | Yes (host file) | No |
| **Portable** | Yes | No (host-path dependent) | N/A |
| **Performance** | Good (native Linux) | Good (host FS) | Best (in-memory) |
| **Use case** | DB data, app state | Dev: hot reload, config injection | Secrets, temp scratch space |

**Best practices:**
- Use **named volumes** for stateful production workloads
- Use **bind mounts** during development for live code reload
- Use **tmpfs** for sensitive data that must never touch disk

```bash
# Named volume (recommended for production)
docker run -v pgdata:/var/lib/postgresql/data postgres

# Bind mount (development)
docker run -v $(pwd)/src:/app/src myapp

# tmpfs (sensitive secrets)
docker run --tmpfs /run/secrets:rw,noexec,nosuid,size=65536k myapp
```

### Q10. How do you share data between containers? What are the trade-offs?

**Answer:**

**Method 1: Named Volumes (recommended)**
```bash
docker volume create shared-data
docker run -v shared-data:/data writer
docker run -v shared-data:/data reader
```
Pros: Docker-managed, portable, works across restarts.
Cons: Concurrent writes need application-level locking.

**Method 2: Bind mounts to a shared host directory**
```bash
docker run -v /host/shared:/data writer
docker run -v /host/shared:/data reader
```
Pros: Easy to inspect from host.
Cons: Host-path coupling, not portable.

**Method 3: Volumes from (`--volumes-from`)** *(legacy)*
```bash
docker run --name data-container -v /data busybox
docker run --volumes-from data-container app
```
Pros: Simple.
Cons: Creates tight coupling; avoid in new designs.

**Method 4: Sidecar container via shared volume**
Common in Kubernetes-style patterns: a sidecar writes logs to a shared volume, and a log shipper container reads them.

## Security

### Q11. What are the main security risks of running containers as root, and how do you mitigate them?

**Answer:**

**Risks of running as root:**
- If a container breakout occurs, the attacker has root on the host
- Writable files owned by root may affect host-mounted volumes
- Increases blast radius of any vulnerability in the app

**Mitigations:**

**1. Use a non-root user in the Dockerfile:**
```dockerfile
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser
USER appuser
```

**2. Use rootless Docker:**
Entire Docker daemon runs as an unprivileged user. Requires kernel user namespace support.
```bash
dockerd-rootless-setuptool.sh install
```

**3. Drop Linux capabilities:**
```bash
docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE nginx
```

**4. Use `--read-only` filesystem:**
```bash
docker run --read-only --tmpfs /tmp myapp
```

**5. Apply Seccomp profiles:**
```bash
docker run --security-opt seccomp=profile.json myapp
```

**6. Use distroless or scratch base images** — no shell = no shell injection attack surface.

### Q12. Explain Docker Content Trust (DCT) and image signing.

**Answer:**

**Docker Content Trust (DCT)** uses **The Update Framework (TUF)** and **Notary** to cryptographically sign and verify Docker images, ensuring authenticity and integrity.

**How it works:**
1. Publisher signs the image with a private key upon push
2. Signature and metadata are stored in a Notary server
3. Consumer enables DCT; Docker verifies the signature before pulling

**Enable DCT:**
```bash
export DOCKER_CONTENT_TRUST=1

# Now push signs automatically
docker push myrepo/myimage:1.0

# Pull verifies signature; fails if unsigned or tampered
docker pull myrepo/myimage:1.0
```

**Key types:**
- **Root key** — offline, highest trust; signs all other keys
- **Repository key (tagging key)** — per-repo; used to sign tags
- **Snapshot key** — signs the current state of the repo
- **Timestamp key** — prevents replay attacks; has short TTL

**In production:** Combine DCT with image scanning (Trivy, Snyk) and private registries for a defense-in-depth approach.

### Q13. What is the difference between `--privileged` and `--cap-add`? When should each be used?

**Answer:**

**`--privileged`:**
- Disables all security mechanisms: seccomp, AppArmor, capabilities restrictions
- Gives container nearly full host kernel access
- Mounts host device filesystem
- **Never use in production** unless absolutely unavoidable (e.g., Docker-in-Docker CI)

**`--cap-add` / `--cap-drop`:**
- Grants or removes specific Linux capabilities only
- More surgical and secure than `--privileged`
- Adheres to principle of least privilege

**Common capabilities and use cases:**

| Capability | Use Case |
|---|---|
| `NET_ADMIN` | Modify network interfaces, routing |
| `NET_BIND_SERVICE` | Bind to ports < 1024 as non-root |
| `SYS_PTRACE` | Debugging tools (strace, gdb) |
| `SYS_ADMIN` | Mount filesystems (avoid if possible) |
| `CHOWN` | Change file ownership |

**Recommended pattern:** Start with `--cap-drop=ALL`, then add only what's needed:
```bash
docker run \
  --cap-drop=ALL \
  --cap-add=NET_BIND_SERVICE \
  --cap-add=CHOWN \
  --read-only \
  nginx
```

## Performance & Optimization

### Q14. How do you optimize a Dockerfile for build speed and image size?

**Answer:**

**1. Leverage layer caching — order matters:**
```dockerfile
# BAD: copies source first, invalidates cache on every code change
COPY . .
RUN pip install -r requirements.txt

# GOOD: install dependencies first (changes rarely)
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
```

**2. Combine RUN commands to reduce layers:**
```dockerfile
# BAD: 3 layers
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# GOOD: 1 layer
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl && \
    rm -rf /var/lib/apt/lists/*
```

**3. Use `.dockerignore`:**
```
.git
node_modules
*.log
tests/
.env
```

**4. Use multi-stage builds** (see Q5).

**5. Choose minimal base images:**

| Base Image | Size | Use Case |
|---|---|---|
| `ubuntu:24.04` | ~78MB | General purpose |
| `alpine:3.19` | ~7MB | Lightweight Linux |
| `distroless/base` | ~20MB | Security-focused |
| `scratch` | 0MB | Statically compiled binaries |

**6. Use BuildKit for parallel builds:**
```bash
DOCKER_BUILDKIT=1 docker build .
```

### Q15. How do you configure and tune cgroup resource limits for containers?

**Answer:**

Docker exposes cgroup controls via `docker run` flags:

**CPU:**
```bash
# Limit to 1.5 CPUs (1500 millicores)
docker run --cpus="1.5" myapp

# Pin to specific CPU cores
docker run --cpuset-cpus="0,1" myapp

# CPU shares (relative weight, default 1024)
docker run --cpu-shares=512 myapp   # gets half the priority
```

**Memory:**
```bash
# Hard limit: OOM kill if exceeded
docker run --memory="512m" myapp

# Soft limit: system can reclaim memory under pressure
docker run --memory="512m" --memory-reservation="256m" myapp

# Disable swap
docker run --memory="512m" --memory-swap="512m" myapp
```

**I/O:**
```bash
# Limit disk read/write speed
docker run --device-read-bps /dev/sda:10mb \
           --device-write-bps /dev/sda:10mb myapp
```

**Verify limits applied:**
```bash
docker inspect <container> | jq '.[0].HostConfig | {Memory, CpuQuota, CpuPeriod}'

# Or via cgroup filesystem
cat /sys/fs/cgroup/memory/docker/<container-id>/memory.limit_in_bytes
```

## Docker Compose

### Q16. What is the difference between `depends_on`, health checks, and startup ordering in Compose?

**Answer:**

`depends_on` alone only controls **start order**, not **readiness**. A dependent container starts when the dependency container is running, not when it's healthy or accepting connections.

**Problem:**
```yaml
# WRONG: web may start before db is ready to accept connections
services:
  web:
    depends_on:
      - db
  db:
    image: postgres
```

**Solution: combine with health checks:**
```yaml
services:
  db:
    image: postgres
    environment:
      POSTGRES_PASSWORD: secret
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 10s

  web:
    image: myapp
    depends_on:
      db:
        condition: service_healthy   # waits for health check to pass
```

**Conditions available:**
- `service_started` — default; just waits for container to start
- `service_healthy` — waits for health check to report healthy
- `service_completed_successfully` — waits for a one-shot init container to exit 0

### Q17. Explain Docker Compose networking and how services discover each other.

**Answer:**

By default, Docker Compose creates a **single bridge network** for each project, named `<project>_default`. All services join this network and can discover each other using their **service name as the hostname**.

**Custom networks:**
```yaml
networks:
  frontend:
  backend:

services:
  nginx:
    networks:
      - frontend
  api:
    networks:
      - frontend
      - backend
  db:
    networks:
      - backend   # db is NOT reachable from nginx
```

**Network aliases:**
```yaml
services:
  db:
    image: postgres
    networks:
      backend:
        aliases:
          - database
          - postgres-primary
```

**External networks** (shared between Compose projects):
```yaml
networks:
  shared-net:
    external: true   # must already exist: docker network create shared-net
```

## Production & Orchestration

### Q18. What is Docker Swarm and how does it compare to Kubernetes?

**Answer:**

**Docker Swarm** is Docker's native clustering and orchestration solution. It turns a pool of Docker hosts into a single virtual host with built-in load balancing, service discovery, and rolling updates.

| Feature | Docker Swarm | Kubernetes |
|---|---|---|
| **Learning curve** | Low | High |
| **Setup complexity** | Simple (`docker swarm init`) | Complex |
| **Scaling** | Manual or basic | Auto-scaling (HPA, VPA) |
| **Self-healing** | Yes (service restarts) | Yes (advanced) |
| **Networking** | Overlay networks, VXLAN | CNI plugins (Calico, Cilium) |
| **Storage** | Volume plugins | PersistentVolumes, CSI |
| **Ecosystem** | Smaller | Massive (CNCF) |
| **Production adoption** | Declining | Dominant |

**When to use Swarm:** Small teams, simpler workloads, teams already deep in Docker tooling, rapid prototyping of distributed apps.

**Swarm key concepts:**
- **Manager nodes** — maintain cluster state (Raft consensus), schedule services
- **Worker nodes** — execute tasks (containers)
- **Services** — desired state definition (replicas, image, ports)
- **Tasks** — individual containers running on nodes

```bash
# Initialize swarm
docker swarm init --advertise-addr <manager-ip>

# Deploy a stack
docker stack deploy -c docker-compose.yml myapp

# Scale a service
docker service scale myapp_api=5
```

### Q19. How do you implement zero-downtime deployments with Docker?

**Answer:**

**1. Rolling Updates (Docker Swarm):**
```bash
docker service update \
  --image myapp:2.0 \
  --update-parallelism 2 \
  --update-delay 10s \
  --update-failure-action rollback \
  myapp_web
```

**2. Health check-gated rollouts:**
```yaml
services:
  web:
    image: myapp:2.0
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost/health"]
      interval: 5s
      retries: 3
    deploy:
      update_config:
        parallelism: 1
        delay: 10s
        failure_action: rollback
        order: start-first   # new container starts BEFORE old one stops
```

**3. Blue-Green Deployment pattern:**
```bash
# Blue is live on port 80
docker run -d --name blue -p 80:8080 myapp:1.0

# Start green
docker run -d --name green -p 8081:8080 myapp:2.0

# Test green, then switch traffic (update load balancer/reverse proxy)
# Update nginx/traefik upstream to point to green:8080

# Remove blue
docker stop blue && docker rm blue
```

**4. Graceful shutdown in application code:**
Handle `SIGTERM` in your app, stop accepting new requests, finish in-flight requests, then exit cleanly.

## Debugging & Troubleshooting

### Q20. What are your go-to commands and techniques for debugging a misbehaving container?

**Answer:**

**1. Inspect container state and config:**
```bash
docker inspect <container>              # full JSON config, mounts, network
docker inspect --format '{{.State}}' <container>
docker inspect --format '{{.HostConfig.Memory}}' <container>
```

**2. View logs:**
```bash
docker logs <container>                 # all logs
docker logs --tail 100 -f <container>  # follow last 100 lines
docker logs --since 5m <container>     # last 5 minutes
```

**3. Execute into running container:**
```bash
docker exec -it <container> /bin/sh    # attach shell
docker exec <container> env            # check environment variables
docker exec <container> cat /etc/hosts # check networking
```

**4. Run ephemeral debug container:**
```bash
# Attach to a container's network namespace
docker run -it --network container:<name> nicolaka/netshoot

# Attach to a container's PID namespace
docker run -it --pid container:<name> busybox
```

**5. Check resource usage:**
```bash
docker stats                            # live CPU/mem/net/IO
docker top <container>                  # processes inside container
```

**6. Copy files in/out:**
```bash
docker cp <container>:/var/log/app.log ./app.log
docker cp ./config.json <container>:/app/config.json
```

**7. Inspect image layers for unexpected content:**
```bash
docker history myimage
dive myimage      # third-party tool for interactive layer inspection
```

**8. Network debugging:**
```bash
docker network inspect <network>
docker run --rm --network container:<name> nicolaka/netshoot \
  nslookup db
```

### Q21. How do you handle secrets management in Docker? What should you avoid?

**Answer:**

**What to AVOID:**
```dockerfile
# NEVER: secrets visible in image history
ENV DB_PASSWORD=supersecret

# NEVER: secrets baked into image layers
RUN curl -H "Authorization: Bearer $TOKEN" https://api.example.com
```

**Correct approaches:**

**1. Docker Secrets (Swarm mode):**
```bash
echo "mysecretpassword" | docker secret create db_password -

docker service create \
  --secret db_password \
  --env DB_PASSWORD_FILE=/run/secrets/db_password \
  myapp
```
Secrets are mounted as in-memory tmpfs files at `/run/secrets/<name>`.

**2. BuildKit secret mounts (build-time secrets):**
```dockerfile
# Secret never stored in any image layer
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc npm install
```
```bash
docker buildx build --secret id=npmrc,src=$HOME/.npmrc .
```

**3. External secrets managers:**
- **HashiCorp Vault** — inject secrets at runtime via sidecar or init container
- **AWS Secrets Manager / Parameter Store** — fetch secrets in entrypoint script
- **Kubernetes Secrets** — when using k8s, use sealed secrets or external-secrets operator

**4. Environment variable files (development only):**
```bash
docker run --env-file .env.local myapp   # never commit .env.local
```

### Q22. How do you debug a container that crashes immediately on startup?

**Answer:**

When a container exits before you can `exec` into it, use these strategies:

**1. Check exit code and logs:**
```bash
docker ps -a                        # see exit code
docker logs <container-id>          # check output before crash
```

**Common exit codes:**
| Code | Meaning |
|---|---|
| 0 | Clean exit |
| 1 | Application error |
| 125 | Docker daemon error |
| 126 | Command not executable |
| 127 | Command not found |
| 137 | OOM killed (128 + SIGKILL) |
| 143 | Graceful termination (128 + SIGTERM) |

**2. Override the entrypoint to inspect the image:**
```bash
docker run -it --entrypoint /bin/sh myimage
docker run -it --entrypoint /bin/sh myimage -c "ls /app && env"
```

**3. Use `--restart=no` to prevent restart loops:**
```bash
docker run --restart=no myimage
```

**4. Inspect resource limits — OOM kills:**
```bash
docker inspect <container> | jq '.[0].State.OOMKilled'
dmesg | grep -i "oom\|killed"
```

**5. Check mount/volume issues:**
```bash
docker run -it --entrypoint sh myimage -c "ls -la /data"
```
