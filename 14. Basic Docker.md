## Core Concepts

### Q1. What is Docker and why is it used?

**Answer:**

Docker is an open-source platform that allows you to **package, ship, and run applications inside containers**. A container bundles the application code together with all its dependencies, libraries, and configuration so it runs consistently on any machine.

**Why use Docker?**

- **"Works on my machine" problem solved** — containers run the same everywhere (dev, staging, production)
- **Lightweight** — containers share the host OS kernel, unlike virtual machines which each need a full OS
- **Fast startup** — containers start in seconds, VMs take minutes
- **Easy to share** — package your app as an image and anyone can run it with one command
- **Isolation** — each container is isolated from others and from the host

**Simple analogy:** Think of a Docker image like a shipping container. It holds everything needed and can be loaded onto any ship (server) without modification.

---

### Q2. What is the difference between a Docker image and a Docker container?

**Answer:**

| | Docker Image | Docker Container |
|---|---|---|
| **What it is** | A read-only template/blueprint | A running instance of an image |
| **State** | Static (does not change) | Dynamic (has a running process) |
| **Analogy** | A class in programming | An object (instance of a class) |
| **Storage** | Stored on disk | Lives in memory + thin writable layer |
| **Created by** | `docker build` or `docker pull` | `docker run` |

```bash
# Image: the blueprint
docker pull nginx                  # download image

# Container: a running instance of that image
docker run nginx                   # create and start a container from the image
docker run nginx                   # creates a SECOND, separate container
```

You can run **many containers** from the **same image**, and each is fully independent.

### Q3. How is Docker different from a Virtual Machine (VM)?

**Answer:**

| Feature | Docker Container | Virtual Machine |
|---|---|---|
| **OS** | Shares host OS kernel | Has its own full OS |
| **Size** | Megabytes | Gigabytes |
| **Startup time** | Seconds | Minutes |
| **Isolation** | Process-level isolation | Full hardware-level isolation |
| **Performance** | Near-native | Overhead from hypervisor |
| **Portability** | Highly portable | Less portable |

**Architecture comparison:**

```
Docker                          Virtual Machine
──────────────────────          ──────────────────────
  App A  |  App B               App A   |   App B
─────────────────────          ─────────────────────
Docker Engine                  Guest OS | Guest OS
─────────────────────          ─────────────────────
    Host OS                      Hypervisor (VMware/VBox)
─────────────────────          ─────────────────────
    Hardware                        Host OS
                               ─────────────────────
                                    Hardware
```

**When to use VMs:** When you need complete OS isolation, different kernels, or strict security boundaries.
**When to use Docker:** When you want fast, lightweight, portable application packaging.

### Q4. What is Docker Hub?

**Answer:**

**Docker Hub** is Docker's official **cloud-based registry** where you can store, share, and distribute Docker images. It's like GitHub but for Docker images.

**Key features:**
- Hosts millions of public images (nginx, postgres, python, node, etc.)
- Free public repositories; paid private repositories
- Official images are maintained and verified by Docker or the software vendor
- Supports automated builds triggered by code pushes

```bash
# Pull (download) an image from Docker Hub
docker pull ubuntu:24.04

# Push (upload) your own image to Docker Hub
docker login
docker tag myapp myusername/myapp:1.0
docker push myusername/myapp:1.0
```

**Other registries:** Amazon ECR, Google Artifact Registry, GitHub Container Registry, self-hosted (Harbor).

---

### Q5. What is a Docker registry?

**Answer:**

A **Docker registry** is a storage and distribution system for Docker images. Docker Hub is the default public registry, but you can use or host your own.

**Types:**
- **Public registry** — anyone can pull images (e.g., Docker Hub)
- **Private registry** — restricted access, used for proprietary or internal images
- **Self-hosted** — run your own registry with Docker's open-source `registry` image

```bash
# Run a local private registry
docker run -d -p 5000:5000 --name registry registry:2

# Tag and push to local registry
docker tag myapp localhost:5000/myapp:1.0
docker push localhost:5000/myapp:1.0

# Pull from local registry
docker pull localhost:5000/myapp:1.0
```

## Images

### Q6. What is a base image?

**Answer:**

A **base image** is the starting point of your Docker image, specified in the `FROM` instruction in a Dockerfile. Every image is built on top of a base image.

**Common base images:**

| Base Image | Description |
|---|---|
| `ubuntu` | Full Ubuntu Linux |
| `alpine` | Tiny Linux (~5MB), great for small images |
| `debian` | Debian Linux, widely used |
| `node` | Node.js pre-installed |
| `python` | Python pre-installed |
| `scratch` | Empty image, for statically compiled binaries |

```dockerfile
# Using ubuntu as a base
FROM ubuntu:24.04

# Using a minimal alpine base
FROM alpine:3.19

# Using an official language image
FROM python:3.12-slim
```

The `slim` and `alpine` variants are stripped-down versions that result in smaller final images.

### Q7. How do you pull and list Docker images?

**Answer:**

```bash
# Pull an image from Docker Hub
docker pull nginx                        # pulls the "latest" tag
docker pull nginx:1.25                   # pulls a specific version
docker pull ubuntu:24.04

# List all images on your machine
docker images
docker image ls

# Example output:
# REPOSITORY   TAG       IMAGE ID       CREATED        SIZE
# nginx        latest    a6bd71f48f68   2 weeks ago    187MB
# ubuntu       24.04     35a88802559d   4 weeks ago    78.1MB

# Remove an image
docker rmi nginx
docker image rm nginx:1.25

# Remove all unused images
docker image prune
```

### Q8. What is an image tag and why is it important?

**Answer:**

An image **tag** is a label attached to a specific version of an image. It's added after a colon: `image:tag`.

```bash
nginx:latest      # most recent version (default if no tag specified)
nginx:1.25        # specific version
python:3.12-slim  # version + variant
ubuntu:24.04      # LTS release
```

**Why tags matter:**

- `latest` is **not automatically the most stable** — it's just the most recently pushed tag
- In production, always pin to a specific tag to ensure reproducibility
- Without a tag, Docker defaults to `latest`, which can break your app when the image updates

```dockerfile
# BAD: unpredictable — could change any time
FROM python:latest

# GOOD: predictable and reproducible
FROM python:3.12.3-slim
```

### Q9. How do you build a Docker image?

**Answer:**

Use the `docker build` command, pointing it at a directory that contains a `Dockerfile`.

```bash
# Basic build (. means use current directory)
docker build .

# Build and tag the image
docker build -t myapp:1.0 .

# Build with a specific Dockerfile name/path
docker build -f Dockerfile.prod -t myapp:prod .

# Build with a build argument
docker build --build-arg ENV=production -t myapp .
```

**What happens during build:**
1. Docker reads the `Dockerfile` top to bottom
2. Each instruction creates a new **layer**
3. Layers are cached — unchanged layers are reused on subsequent builds
4. The final result is a new image stored locally

```bash
# After building, verify it exists
docker images | grep myapp
```

## Containers

### Q10. How do you run, stop, and remove a container?

**Answer:**

```bash
# Run a container
docker run nginx                             # run in foreground
docker run -d nginx                          # run in background (detached)
docker run -d -p 8080:80 --name web nginx    # with port mapping and name

# List running containers
docker ps

# List ALL containers (including stopped)
docker ps -a

# Stop a container (sends SIGTERM, then SIGKILL after 10s)
docker stop web

# Start a stopped container
docker start web

# Restart a container
docker restart web

# Remove a stopped container
docker rm web

# Force remove a running container
docker rm -f web

# Run and auto-remove when it exits
docker run --rm ubuntu echo "hello"
```

---

### Q11. What is the difference between `docker run` and `docker start`?

**Answer:**

| | `docker run` | `docker start` |
|---|---|---|
| **Purpose** | Creates AND starts a new container from an image | Starts an existing stopped container |
| **Requires** | An image name | A container name or ID |
| **Creates new container?** | Yes, every time | No, reuses the existing one |

```bash
# docker run: creates a brand new container each time
docker run -d --name app nginx      # creates container "app"
docker run -d --name app nginx      # ERROR: name already taken

# docker start: restarts the stopped "app" container
docker stop app
docker start app
```

### Q12. How do you view container logs?

**Answer:**

```bash
# View all logs from a container
docker logs <container-name-or-id>

# Follow logs in real time (like tail -f)
docker logs -f web

# Show last N lines
docker logs --tail 50 web

# Show logs with timestamps
docker logs -t web

# Show logs since a time
docker logs --since 10m web        # last 10 minutes
docker logs --since "2024-01-15"   web
```

Logs come from the container's **stdout and stderr**. This means your application should write logs to stdout, not to log files inside the container.

### Q13. How do you execute a command inside a running container?

**Answer:**

Use `docker exec` to run a command inside an already-running container.

```bash
# Open an interactive shell
docker exec -it web /bin/bash
docker exec -it web /bin/sh        # use sh if bash is not available

# Run a single command
docker exec web ls /usr/share/nginx/html
docker exec web env                # print environment variables
docker exec web cat /etc/nginx/nginx.conf

# Run as a specific user
docker exec -u root -it web bash
```

**Flags:**
- `-i` — keep STDIN open (interactive)
- `-t` — allocate a pseudo-TTY (terminal)
- `-u` — specify user

## Dockerfile

### Q14. What is a Dockerfile and what are the most common instructions?

**Answer:**

A **Dockerfile** is a plain text file with step-by-step instructions that Docker uses to automatically build an image. Each instruction creates one layer in the image.

**Most common instructions:**

| Instruction | Purpose | Example |
|---|---|---|
| `FROM` | Set base image (required, must be first) | `FROM node:20-alpine` |
| `WORKDIR` | Set working directory inside the image | `WORKDIR /app` |
| `COPY` | Copy files from host into image | `COPY . .` |
| `ADD` | Like COPY, but also handles URLs and .tar | `ADD app.tar.gz /app` |
| `RUN` | Execute command during build | `RUN npm install` |
| `ENV` | Set environment variable | `ENV NODE_ENV=production` |
| `EXPOSE` | Document which port the app uses | `EXPOSE 3000` |
| `CMD` | Default command when container starts | `CMD ["node", "server.js"]` |
| `ENTRYPOINT` | Fixed executable for the container | `ENTRYPOINT ["python"]` |
| `ARG` | Build-time variable (not in final image) | `ARG VERSION=1.0` |
| `VOLUME` | Declare mount point for volumes | `VOLUME /data` |

---

### Q15. Write a simple Dockerfile for a Node.js application.

**Answer:**

```dockerfile
# 1. Start from an official Node.js image
FROM node:20-alpine

# 2. Set the working directory inside the container
WORKDIR /app

# 3. Copy package files first (for layer caching)
COPY package.json package-lock.json ./

# 4. Install dependencies
RUN npm install --production

# 5. Copy the rest of the application code
COPY . .

# 6. Document the port the app listens on
EXPOSE 3000

# 7. Command to run when the container starts
CMD ["node", "server.js"]
```

**Build and run:**
```bash
docker build -t my-node-app:1.0 .
docker run -d -p 3000:3000 --name myapp my-node-app:1.0
# Visit http://localhost:3000
```

### Q16. What is `.dockerignore` and why should you use it?

**Answer:**

A `.dockerignore` file tells Docker which files and directories to **exclude** when copying files into the image with `COPY` or `ADD`. It works just like `.gitignore`.

**Why it matters:**
- Keeps your image small by excluding unnecessary files
- Speeds up builds (less data sent to Docker daemon)
- Prevents secrets or sensitive files from leaking into images

**Example `.dockerignore`:**
```
# Dependencies (reinstalled inside container)
node_modules/

# Version control
.git/
.gitignore

# Environment and secrets
.env
.env.local

# Logs and temp files
*.log
tmp/

# Test files
tests/
coverage/

# Documentation
README.md
docs/
```

### Q17. What is the difference between `COPY` and `ADD` in a Dockerfile?

**Answer:**

| | `COPY` | `ADD` |
|---|---|---|
| **Basic file copy** | ✅ Yes | ✅ Yes |
| **Copy directories** | ✅ Yes | ✅ Yes |
| **Auto-extract `.tar` files** | ❌ No | ✅ Yes |
| **Download from URL** | ❌ No | ✅ Yes |
| **Recommended for?** | Most use cases | Only when extraction is needed |

```dockerfile
# COPY: straightforward and predictable
COPY ./src /app/src
COPY package.json .

# ADD: use only when you need tar extraction
ADD app.tar.gz /app/        # extracts the archive automatically

# Downloading via ADD is discouraged — use RUN curl instead
# because it creates fewer layers and you can clean up in the same step
RUN curl -O https://example.com/file.tar.gz && \
    tar -xzf file.tar.gz && \
    rm file.tar.gz
```

**Best practice:** Use `COPY` by default. Only use `ADD` when you need automatic archive extraction.

## Networking Basics

### Q18. What is port mapping in Docker?

**Answer:**

By default, a container's ports are **not accessible from outside** the Docker network. **Port mapping** (also called port publishing) connects a port on the **host machine** to a port inside the **container**.

```bash
# Syntax: -p <host-port>:<container-port>
docker run -p 8080:80 nginx
#                ^    ^
#          host port  container port
# Access via http://localhost:8080 → nginx inside container on port 80

# Multiple port mappings
docker run -p 8080:80 -p 443:443 nginx

# Bind to a specific host IP (only accessible locally)
docker run -p 127.0.0.1:8080:80 nginx

# Random available host port
docker run -p 80 nginx
docker port <container>    # find which host port was assigned
```

`EXPOSE` in a Dockerfile is **documentation only** — it does not publish the port. You must use `-p` when running the container.

---

### Q19. What Docker network types exist at a basic level?

**Answer:**

The three most common networks you'll encounter:

**1. bridge (default)**
- Containers on the same bridge network can talk to each other
- Requires port mapping to expose to the host
- Default network for `docker run`

```bash
docker run -d --name app1 myapp
docker run -d --name app2 myapp
# app1 and app2 are on the default bridge network
```

**2. host**
- Container uses the host machine's network directly
- No port mapping needed (and `-p` is ignored)
- Less portable, Linux only

```bash
docker run --network host nginx
# nginx is now accessible on host's port 80 directly
```

**3. none**
- Container has no network access at all
- Fully isolated

```bash
docker run --network none myapp
```

## Volumes & Storage

### Q20. What is a Docker volume and why do you need one?

**Answer:**

By default, any data written inside a container is **lost when the container is removed**. A **volume** is a way to store data **outside the container's lifecycle** so it persists.

```bash
# Without a volume: data lost when container removed
docker run postgres
docker rm postgres      # all database data GONE

# With a named volume: data persists
docker run -v pgdata:/var/lib/postgresql/data postgres
docker rm postgres
docker run -v pgdata:/var/lib/postgresql/data postgres   # data still there!
```

**Common use cases for volumes:**
- Database files (MySQL, PostgreSQL, MongoDB)
- Uploaded user files
- Application configuration
- Log files that need to outlive the container

```bash
# Create a named volume
docker volume create mydata

# List volumes
docker volume ls

# Inspect a volume (see where data is stored on host)
docker volume inspect mydata

# Remove a volume
docker volume rm mydata

# Remove all unused volumes
docker volume prune
```

---

### Q21. What is the difference between a volume and a bind mount?

**Answer:**

| | Volume | Bind Mount |
|---|---|---|
| **Managed by** | Docker | You (host filesystem) |
| **Host path** | Docker chooses (`/var/lib/docker/volumes/`) | You specify (any path) |
| **Use case** | Production data persistence | Development, config files |
| **Portable** | Yes | No (depends on host path) |

```bash
# Named volume (Docker manages the location)
docker run -v mydata:/app/data myapp

# Bind mount (you specify host path)
docker run -v /home/user/myproject:/app myapp
docker run -v $(pwd):/app myapp          # current directory

# Bind mount is useful during development:
# changes to host files instantly appear in the container
docker run -v $(pwd)/src:/app/src -p 3000:3000 myapp
```

## Docker Compose Basics

### Q22. What is Docker Compose and when would you use it?

**Answer:**

**Docker Compose** is a tool for defining and running **multi-container Docker applications**. Instead of running many `docker run` commands manually, you describe all services in a single `docker-compose.yml` file and start everything with one command.

**When to use it:**
- Your app needs multiple services (web server + database + cache)
- You want a reproducible local development environment
- You want to share your environment setup with teammates

**Example: web app + database:**

```yaml
# docker-compose.yml
version: "3.9"

services:
  web:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgres://user:pass@db:5432/mydb
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

```bash
docker compose up           # start all services
docker compose up -d        # start in background
docker compose down         # stop and remove containers
docker compose logs -f      # follow all logs
docker compose ps           # list running services
```

---

### Q23. What is the structure of a `docker-compose.yml` file?

**Answer:**

A `docker-compose.yml` has four top-level sections:

```yaml
version: "3.9"         # Compose file format version

services:              # Define each container (required)
  service-name:
    image: nginx       # use an existing image
    build: .           # OR build from Dockerfile
    ports:
      - "host:container"
    environment:
      - KEY=VALUE
    volumes:
      - named-vol:/path
    depends_on:
      - other-service
    restart: always

volumes:               # Declare named volumes (optional)
  named-vol:

networks:              # Declare custom networks (optional)
  my-network:
```

**Common `restart` policies:**

| Policy | Behavior |
|---|---|
| `no` | Never restart (default) |
| `always` | Always restart |
| `on-failure` | Restart only on non-zero exit |
| `unless-stopped` | Always restart unless manually stopped |

## Common Commands

### Q24. What are the most essential Docker commands every developer should know?

**Answer:**

**Images:**
```bash
docker pull <image>              # download image
docker images                    # list local images
docker build -t name:tag .       # build image from Dockerfile
docker rmi <image>               # remove image
docker image prune               # remove unused images
```

**Containers:**
```bash
docker run -d -p 8080:80 --name web nginx   # run container
docker ps                                    # list running containers
docker ps -a                                 # list all containers
docker stop <name>                           # stop container
docker start <name>                          # start stopped container
docker rm <name>                             # remove container
docker rm -f <name>                          # force remove running container
```

**Inspection & Debugging:**
```bash
docker logs <name>               # view container logs
docker logs -f <name>            # follow logs
docker exec -it <name> sh        # open shell in container
docker inspect <name>            # detailed container info
docker stats                     # live resource usage
docker top <name>                # processes inside container
```

**Volumes & Networks:**
```bash
docker volume ls                 # list volumes
docker volume create mydata      # create volume
docker volume rm mydata          # remove volume
docker network ls                # list networks
docker network create mynet      # create network
```

**System Cleanup:**
```bash
docker system df                 # show disk usage
docker system prune              # remove all unused resources
docker container prune           # remove stopped containers
docker image prune -a            # remove all unused images
```

### Q25. What does `docker inspect` show and when is it useful?

**Answer:**

`docker inspect` returns a **detailed JSON object** with the full configuration and state of a container, image, volume, or network.

```bash
docker inspect web               # inspect container named "web"
docker inspect nginx             # inspect an image
docker inspect myvolume          # inspect a volume
```

**Useful things you can find:**
- Container IP address
- Environment variables
- Volume mounts
- Port mappings
- Restart policy
- Whether it's running and its exit code
- Network settings

**Extract specific fields with `--format`:**
```bash
# Get container IP address
docker inspect --format '{{.NetworkSettings.IPAddress}}' web

# Get the image used
docker inspect --format '{{.Config.Image}}' web

# Check if running
docker inspect --format '{{.State.Running}}' web
```
